## 0. 발표 전략 한 장 요약

### 중간발표 → 최종발표, 무엇이 달라지는가
- **중간발표**: "이런 시스템을 제안한다" — 문제 인식 + 방법론 + 4단계 파이프라인 + CleanGo 워크스루 (전부 *개념과 설계*)
- **최종발표의 핵심 추가분**: "그 방법론이 **정말 효과가 있는가**를 정량 검증했다" — LLM-as-a-judge A/B 평가, 8차원 32항목, effort 스윕 + stage ablation, 그리고 **"이득이 어디서 오는가"** 에 대한 정량·정성 통찰

### 발표의 한 줄 메시지
> 단계형 Human-in-the-loop 기획 에이전트는 일회성 LLM 생성을 **24쌍 중 22~23번 이긴다(p<0.0001)**. 그리고 그 우위의 본질은 *단계 수*가 아니라 **거버넌스·측정·실행 구조를 강제하는 합성 레이어**에 있다.

### 시간/분량 가이드
- 총 23슬라이드(섹션 표지 포함), 발표 15~20분 기준
- 비중: **PART 1 개요(recap) 25% / PART 2 검증(신규 핵심) 45% / PART 3 통찰·결론 30%**
- 시간이 짧으면 S5(CleanGo 상세), S11(편향 통제), S20(로드맵)을 압축/생략 가능

### 발표 톤 (기술혁신 수업 맥락)
- 단순 ML 평가 보고가 아니라 **"기획→개발 번역 병목"이라는 혁신 과제를, 구조화된 인간-AI 협업(process innovation)으로 푸는 시도 + 그 효과의 실증**으로 프레이밍
- 가장 차별화되는 지점 2개를 반드시 부각: ① **순환 논증을 피한 공정한 평가 설계**(형식이 아니라 내용을 측정), ② **반직관적 발견**(단계 수보다 구조 강제가 핵심)

## PART 1 — 시스템 개요 (recap, 압축)

> 목적: 처음 보는 평가자도 PART 2를 이해할 만큼만 빠르게 재정렬. 새 내용 추가 금지, 중간발표 자산 재사용.

### S3. 문제: 기획자와 개발자 사이의 번역 병목
- 핵심 한 줄: 기획자의 **"무엇을 만들자"** 와 개발자의 **"어떻게 만들까"** 사이 간극 → 아이디어를 구현 가능한 기술기획서로 번역하는 과정이 **조직 자원의 막대한 소모**
- 양쪽 현실 2단(중간발표 S4 압축):
  - 기획자: 기술 제약 지식 부족 / 데이터·규제 판단 어려움 / "일단 MVP로" 반복
  - 개발자: 실현가능성 미검토 요구 수신 / KPI 없는 스펙 / "왜 필요한지" 맥락 소실
- 마무리 명제: **아이디어를 기술 언어로 구체화하는 과정이 체계 없이 개인 경험에만 의존하는 것이 병목**
- 🗣️ 혁신 프레이밍: "이건 도구 부재가 아니라 *프로세스의 부재*다. 우리는 이 번역 과정을 구조화한다."
- 🖼️ 비주얼: 중간발표 S4의 2단 카드 재사용

### S4. Milemate가 하는 일 + 4단계 파이프라인 (개요 1장)
- 정의: 기획자의 **자연어 아이디어**를 개발자가 바로 착수 가능한 **기술기획서(PRD)** 로 번역하는 보조 에이전트 시스템
- 3대 축(중간발표 S5): ① **구조화** ② **범위 확정(MVP)** ③ **검증과 기록(HITL)**
- 4단계 파이프라인 + 멀티에이전트(중간발표 S6 다이어그램 재사용):
  1. **문제정의·KPI 정렬** (Orchestrator, Goal Aligner)
  2. **서비스 구조·MVP 범위** (Solution Architect)
  3. **기술/데이터/규제 검증** (Tech Verifier, Data Analyst, Regulatory Auditor, *3 agent 병렬*)
  4. **최종 리포트** (Spec Writer, Decision Logger) → Planner Report / Engineer Report / Decision Log
- 핵심 원리 강조: **각 단계는 사용자 승인 후 진행**, 문제 발견 시 **이전 단계로 rollback** (Human-in-the-loop)
- 🖼️ 비주얼: 중간발표 S6 파이프라인 다이어그램 그대로 재사용
- 🗣️ 멘트: 이 4단계가 PART 2에서 검증·해부할 대상임을 예고

### S5. (선택) CleanGo로 본 한 단계의 산출물 — "기획서가 실제로 이렇게 나온다"
- 목적: 평가 대상이 추상이 아니라 **구체적 기획 문서**임을 한 장으로 각인
- CleanGo(라스트마일 세탁 물류 스타트업) 예시로, Stage 1 입력(자연어) → Stage 4 산출(Planner/Engineer Report + Decision Log)의 변환을 압축 제시
- 중간발표 Stage1~4 슬라이드 중 **가장 임팩트 있는 1~2개 산출물 캡처**만 발췌
- ⚠️ 시간 부족 시 생략 가능 (S4에서 충분히 전달되면)
- 🖼️ 비주얼: 중간발표 Stage1(UI 목업) + Stage4(리포트 분기) 캡처 발췌

### S6. 전환 슬라이드: 제안에서 검증으로
- 큰 문구: **"그래서 — 이 단계들이 정말 더 나은 기획서를 만드는가?"**
- 중간발표는 *효용을 주장*했지만 증거가 없었음을 솔직히 인정 → 최종발표는 이를 **정량 검증**
- 🗣️ 멘트: "여기서부터가 중간발표 이후 새로 한 일입니다." (PART 2 진입 선언)

## PART 2 — 방법론 유효성 검증 ⭐ (최종발표 핵심)

### S7. 연구 질문 (Research Questions)
- **RQ1.** 4단계 파이프라인(B)이 동일 입력의 일회성 생성(A)보다 더 나은 기획서를 만드는가?
- **RQ2.** 그 이득은 **어느 단계 / 어떤 속성**에서 오는가? (단계를 늘릴수록 좋아지는가?)
- **RQ3.** 결과가 모델의 **reasoning effort**에 좌우되는가? (effort만 높이면 되는 것 아닌가?)
- 🗣️ 멘트: RQ2가 이 발표에서 가장 흥미로운 답을 주는 질문임을 예고

### S8. 실험 설계 — A vs B
| 조건 | 정의 |
|---|---|
| **A (Baseline)** | 동일 initial_input을 주고 **중간 단계 없이 한 번에** 최종 기획서 생성 |
| **B (Proposed)** | 동일 input으로 **4단계(문제정의→MVP범위→리스크검증→최종)** 를 거쳐 생성 |
- 통제: **동일 생성 모델 · 동일 토큰 예산 상한** (단계 누적으로 길어지는 건 허용하되 길이 보정 적용)
- 표본: **8개 라스트마일 시나리오 × seed ≥ 3쌍** (분산 통제)
- 🖼️ 비주얼(신규): **A 1-arm vs B 4-arm** 양갈래 흐름 다이어그램 (같은 입력 → 두 경로 → 최종 문서쌍 → 판정자). *데이터 기반 아님, 단순 도식 → 제작 필요*
- 🗣️ 멘트: "공정성을 위해 모델·입력·토큰을 똑같이 맞췄습니다."

### S9. 평가 척도 — LLM-as-a-judge, 8차원 32항목
- 판정 방식: 두 문서를 익명(문서1/문서2)으로 제시, **항목마다 pass(1.0)/partial(0.5)/fail(0.0)/na**, **모든 pass·partial에 원문 인용 필수**
- 차원 점수 = 1 + 4 × (획득 점수 합 / 유효 항목 수) — *판정자가 아니라 파이프라인이 산식으로 계산(산수 오류 방지)*
- **8개 차원(각 3~5항목, 총 32항목):**
  - D1 문제 정의 품질 / D2 범위 규율(MVP) / D3 실행 구체성 / D4 측정 가능성
  - D5 리스크 식별·보류 조건 / D6 내적 일관성 / D7 의사결정·인계 준비도 / D8 입력 충실도·절제
- 🖼️ 비주얼(신규): 8차원을 도는 휠/그리드 + 대표 항목 1개씩. *제작 필요*

### S10. 평가의 신뢰성 ① — "형식이 아니라 내용을 측정한다" (순환 논증 회피)
- **핵심 공정성 원칙**: 체크리스트는 "좋은 기획서가 갖춰야 할 **내용 속성**"만 측정. 제안 방법론의 **산출 형식(섹션 구조·용어)** 은 측정하지 않음
  - 형식을 기준 삼으면 → "B는 B의 형식을 따르니 B가 이긴다"는 **순환 논증**이 됨
  - 판정자는 섹션 제목이 아니라 **해당 내용이 문서 어디에든 존재하고 구체적인지**만 본다
- **감점 항목 P1~P4** (차원 점수와 별도, 총점 차감):
  - P1 장황함(같은 주장 3회+ 반복, -0.5) / P2 보일러플레이트(시나리오 고유명사 0인 섹션, -0.5/섹션) / P3 근거 없는 정밀 수치(-0.5/건) / P4 입력 제약 위반(-1.0/건)
- 🗣️ 멘트(가장 중요): "이 설계 덕분에 'B가 자기 형식대로 써서 이긴 것 아니냐'는 반박을 차단합니다." — 평가자/교수 질문 선제 대응

### S11. (선택) 평가의 신뢰성 ② — 편향 통제 프로토콜
| 편향 | 통제 |
|---|---|
| 위치 편향 | **통제 안 함, 한계로 명시** (BA 고정 순서 1회 판정으로 비용 절반; `--orders BA,AB`로 2회 스왑 판정 전환 가능) |
| 길이/장황함 | "길이는 품질 아니다" 프롬프트 명시 + P1 + B문서 섹션 간 반복 제거 후 제출 |
| 형식 | 두 문서 동일 plain text로 변환 후 제시 |
| 자기 선호 | 판정 모델을 생성 모델과 분리(또는 다른 temp·프롬프트) 권장 |
| 조건 누출 | "1단계에서…" 등 과정 언급 마스킹, 판정자에게 A/B 미고지 |
| 판정 분산 | 시나리오당 seed ≥ 3 |
- ⚠️ 시간 부족 시 S10에 "편향도 통제했다" 한 줄로 흡수 가능
- 🗣️ 멘트: 위치편향을 *숨기지 않고 한계로 보고*하는 점이 오히려 신뢰도를 높임

### S12. 실험 1 결과 — Reasoning Effort 스윕 (RQ1·RQ3)
- 설정: gpt-5.5, low/medium/high/xhigh 4수준, 생성·판정 동일 effort
- **결과: 모든 effort에서 B 23승 / A 1승 / tie 0**, 전부 **p < 0.0001**

| 지표 | low | medium | high | xhigh |
|---|---|---|---|---|
| B 승 / A 승 | 23 / 1 | 23 / 1 | 23 / 1 | 23 / 1 |
| 평균 점수차 (B−A) | +0.524 | +0.525 | +0.430 | +0.587 |
| 95% CI | [+0.36, +0.71] | [+0.38, +0.67] | [+0.22, +0.66] | [+0.42, +0.78] |

- **메시지: ① B가 A를 압도한다. ② reasoning effort를 올려도 격차가 메워지지 않는다 → 이득은 effort가 아니라 *구조*에서 온다.**
- 🖼️ 비주얼(신규, 데이터 차트): effort별 B/A 승수 막대 + 평균 점수차 점·오차막대. *eval 데이터로 생성 가능*

### S13. 실험 2 결과 — Stage Ablation (RQ2의 핵심 답)
- 설정: gpt-5.5, medium effort. B 조건을 **단계 수로 분해**:
  - stages_1 = 1단계 + 합성 / stages_2 = 1~2단계 + 합성 / stages_3 = 1~3단계 + 합성 / stages_4 = 전체 4단계
- **결과: stages_1~3 = B 22승/A 2승, stages_4 = B 23승/A 1승. 전부 p<0.0001.**

| | stages_1 | stages_2 | stages_3 | stages_4 |
|---|---|---|---|---|
| B 승 / A 승 | 22 / 2 | 22 / 2 | 22 / 2 | 23 / 1 |
| 평균 점수차 (B−A) | +0.634 | +0.716 | +0.591 | +0.723 |

- **⭐ 반직관적 핵심 발견: 1단계(문제정의)만으로도 이미 22승.** 단계를 늘려도 승패는 거의 안 바뀜
- 해석: 우위의 대부분은 *단계 수*가 아니라 **N개 승인 결과를 완전한 PRD로 강제 합성하는 output_layer(구조적 스캐폴딩)** 에서 나온다
- 🖼️ 비주얼(신규, 데이터 차트): stages_1→4 평균 점수차 라인/막대 (거의 평평함을 보여주는 게 포인트). *eval 데이터로 생성 가능*

### S14. 어디서 이기는가 — 차원별 결과
- B가 **압도하는 차원**(거의 100% 승): **D3 실행 구체성, D4 측정 가능성, D7 의사결정·인계 준비도** — 정확히 *구조* 차원
- B가 **확연히 이기는 차원**: D1 문제 정의 (~70%)
- **차이 없는 차원**(둘 다 높아 동률): D2 범위, D5 리스크 — *낮은 "승률"은 열세가 아니라 동률임에 주의*
- **둘 다 취약한 차원**: **D8 입력 충실도·절제** (특히 D8-2 수치 출처) → PART 3에서 심화
- ⚠️ 해석 주의 멘트: "여기서 승률이 낮은 차원은 B가 진 게 아니라, 두 문서 모두 만점이라 *비긴* 것입니다."
- 🖼️ 비주얼(신규, 데이터 차트): 차원별 B 승률 막대 또는 레이더. *데이터로 생성 가능*

### S15. 방법론이 *정확히 무엇을* 추가하는가 — 항목 단위 분석 (the money slide)
- 32항목 중, A(일회성)가 구조적으로 비어 있고 B가 채우는 항목을 대비:

| 항목 | A 평균 | B 평균 | 의미 |
|---|---|---|---|
| **D4-3 KPI 담당 주체** | **0.00** | **+0.96** | A엔 KPI owner 열 자체가 없음 — 단일 최대 격차 |
| **D7-1 결정 안건 완결성** | 0.25 | ~1.00 | A는 선택지 2개+결정주체를 안 붙임 |
| **D7-2 미해결 질문 추적** | 0.27 | ~1.00 | A는 답변 주체·기한을 안 붙임 |
| D3-1/2/3 화면·예외·정책 구조 | ~0.50 | ~1.00 | A는 구조 4요소가 불완전 |
| D1-3 현행 우회 방식·한계 | 0.62 | ~1.00 | A는 "지금 어떻게 버티나"를 자주 누락 |
| **D8-2 수치 출처 추적** | 0.08 | 0.17~0.29 | **둘 다 못 풂** — 파이프라인의 구조적 공백 |

- 메시지: 이득은 *의사결정 거버넌스 · KPI 측정 구조 · 운영 정책 구조* 라는 **구조적 완전성**에 집중됨 → 일회성 GPT엔 자연스럽게 안 나오는 것들
- 🖼️ 비주얼(신규, 데이터 차트, 강력 추천): **A vs B 덤벨(dumbbell)/다이버징 막대** — 항목별 0.0→1.0 점프를 시각적으로. *데이터로 생성 가능*

## PART 3 — 통찰 · 논의 · 결론

### S16. 단계별 기여 분석 — 각 단계가 실제로 더하는 것
- **2단계(서비스 설계) = 가장 안정적인 한계 이득**: 정책 4요소(D3-3), 데이터 필드 명세(D3-4), KPI 구조화(D4) — 시나리오 의존성 낮고 일관적
- **3단계(타당성 검증) = 개념적 신규성 ↑, 품질 mixed**: capacity ledger / 개입 등급(intervention grade) / shadow mode 같은 운영 깊이 추가 ↔ 동시에 근거 없는 수치(D8-2)·내적 비일관(D6)·반복(P1) 유입
- **4단계(최종화) = 시나리오 의존적, 때때로 퇴행**: 일부는 인계 자료를 강화하지만, 문서 목적이 어긋나면(예: PRD→개발 산정 회의자료) 점수 하락
- 메시지: **"단계 추가 = 순이득"이 아니다.** 후기 단계일수록 정교화와 잡음이 함께 늘어남
- 🖼️ 비주얼: 단계 1→2→3→4 위에 ▲(추가)·▼(퇴행) 아이콘을 얹은 흐름도. *간단 도식, 제작 필요*

### S17. 실패 케이스 분석 — A가 이긴 4건의 공통 원인
- 4개 패배 케이스(checkout_fee, rider_onboarding, dispatch, return_pickup)는 **단 하나의 구조적 실패**를 공유:
  - B는 개념 레이어(아키텍처·게이트·정교화)를 쌓으면서 **기존 구조 슬롯(KPI owner, 결정 선택지 2개+, 질문 기한)을 채우지 않고 넘어감**
  - 체크리스트는 미완성 구조를 partial이 아니라 **hard fail** 처리 → 개념적으로 정교해도 *구조가 완전한 단순한 A에게 짐*
- **함의: 파이프라인에 "구조 완전성 감사(completeness audit)" 단계가 없다.** 새 개념을 더할 때 기존 슬롯 충족을 강제하는 메커니즘 부재
- 🗣️ 멘트: 패배조차 *일관된 단일 원인*으로 설명된다는 점이 분석의 강점

### S18. 구조적 공백 — D8-2 수치 출처 추적
- D8-2(정밀 수치의 출처/가정 추적)는 **모든 단계가 체계적으로 못 푸는 공백** (A 0.08, B 최고 0.29)
- 각 단계가 더하는 정밀 수치(신뢰도 임계값·지연 임계값·샘플 하한·차단기)가 **출처·가정 없이** 등장하는 게 패턴
- 유일한 전면 해결 사례 = `cs_repeat_inquiry_triage` 4단계: 수치를 *제거*한 게 아니라 **"현재 미계측; 드라이런으로 조정" 같은 인식론적 레이블**을 붙여 해결 → 이 기법이 다른 시나리오로 전파 안 됨(전용 패스 부재)
- 메시지: 정직한 약점 보고 + 해법 단서 동시 제시
- 🖼️ 비주얼: 텍스트 중심, 8개 시나리오 D8-2 점수 미니 막대(거의 바닥) 가능. *데이터로 생성 가능*

### S19. 종합 해석 — 이 혁신의 가치는 어디에 있나 (innovation lens)
- 정량+정성 종합 결론: **가치는 "단계 수"가 아니라 "구조적 스캐폴딩을 강제하는 것"에 있다.**
  - 일회성 LLM이 *자연스럽게는 안 만드는* 거버넌스·측정·정책 구조를, 단계형 합성이 **반강제로 채워 넣는다**
- 기술혁신 관점 메시지:
  - **Human-in-the-loop + 구조 강제 = 도구가 아니라 프로세스 혁신** — 기획-개발 번역 병목을 *개인 역량 의존*에서 *재현 가능한 구조*로 이동
  - 단, 혁신의 효과는 "더 많은 자동화 단계"가 아니라 **"무엇을 반드시 갖추게 강제하느냐"** 에서 나옴 → 후속 설계 원칙
- 🗣️ 멘트: 발표의 지적 하이라이트. RQ2의 답을 한 문장으로.

### S20. (선택) 개선 로드맵 — 검증이 드러낸 다음 단계
- ① **구조 완전성 감사 단계 추가** (새 개념 추가 시 KPI owner·결정 선택지·질문 기한 충족 강제) — 실패 케이스 4건의 직접 해법
- ② **수치 출처 검증 전용 패스** (D8-2: "미계측/드라이런" 레이블 기법을 전 시나리오로 일반화)
- ③ **가드레일 중복 제거** (output_layer 합성이 안전 제약을 섹션 간 반복 → P1) + **신규 KPI owner 강제**
- ⚠️ 시간 부족 시 S19에 흡수 가능
- 🖼️ 비주얼: 현재 4단계 + 점선으로 "audit / sourcing" 단계 추가하는 다이어그램. *제작 필요*

### S21. 한계 (Limitations) — 정직하게
- **위치 편향**: BA 고정 순서 1회 판정 → B에 유리할 잠재 편향. (`--orders BA,AB` 스왑 2회 판정으로 강화 가능)
- **단일 판정 모델 / 자기 선호**: 생성·판정이 같은 모델 계열이면 자기 선호 가능 → 이종 판정 모델로 재현 필요
- **LLM-judge 타당성**: 사람 평가와의 상관 미검증
- **단일 도메인 8 시나리오**: 라스트마일 한정 — 일반화 주장 제한
- **output_layer의 인위성**: ablation을 위해 추가한 합성 단계 → 실제 제품 단계와 동일하지 않음
- 🗣️ 멘트: 한계를 먼저 말하면 Q&A 방어가 쉬워짐

### S22. 결론
- **한 줄 결론**: 단계형 Human-in-the-loop 기획 에이전트는 일회성 생성을 **유의하게(22~23/24, p<0.0001)** 능가하며, 그 우위의 본질은 **구조적 완전성의 강제**에 있다.
- **두 가지 기여**: ① 기획→개발 번역 병목을 푸는 *시스템* ② 순환 논증을 피한 *공정한 평가 방법론*과 그 결과
- **한 문장 takeaway**: "AI 기획 보조의 핵심은 더 똑똑한 모델이 아니라, **무엇을 반드시 갖추게 강제하는 구조**다."
- 🖼️ 비주얼: PART 1의 파이프라인 다이어그램 위에 검증 결과 숫자를 얹은 마무리 도식(선택)

### S23. 감사합니다 / Q&A
