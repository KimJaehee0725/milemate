# 문제 재정의 메모

## 배경
초기에는 프로젝트를 "기획자의 기술 개발 난이도 조사 어시스턴트"로 정의했다.
그러나 논의 결과, 핵심 문제는 단순히 "이 기술이 어려운가"를 알려주는 것이 아닐 수 있다.

## 새 관점
많은 기획 아이디어는 기술집약적이어서 제품/서비스 기획 자체가 어렵다.
문제는 단순한 개발 난이도 판단보다 다음에 가깝다.

- 아이디어 실현에 어떤 기반 기술이 필요한지 모른다.
- 아이디어를 구현 가능한 기능 단위로 쪼개기 어렵다.
- 어떤 부분이 핵심 기술 의존성이 큰지 파악하기 어렵다.
- 현재 가능한 수준과 아직 어려운 수준을 구분하기 어렵다.
- 처음부터 완성형 서비스를 상상해 범위를 과도하게 크게 잡는다.

즉, 현재 문제는 "기술이 너무 어려움" 자체보다는 "기획 단계에서 필요한 기반 기술과 제약을 구조화하기 어려움"에 더 가깝다.

## 수정된 프로젝트 정의
### 기존 정의
기획자의 아이디어를 받아 개발 난이도와 기술 난제를 분석해주는 에이전트

### 수정 정의
기획자의 아이디어를 받아, 해당 아이디어가 성립하기 위해 필요한 기반 기술을 식별하고, 이를 구현 가능한 기능 단위로 구조화하며, 기술 의존성과 현실적 MVP 범위를 제안하는 기획 보조 에이전트

## 에이전트의 핵심 역할 변화
### 기존 강조점
- 어렵다 / 쉽다 판단
- 난제 정의
- 우회 전략 제시

### 수정 강조점
- 아이디어를 기능 구조로 번역
- 필요한 기반 기술 식별
- 기술 의존성 맵 제시
- 지금 가능한 범위와 어려운 범위 구분
- MVP 수준의 현실적 범위 제안

## 이 변화가 중요한 이유
이렇게 바꾸면 프로젝트가 더 설득력 있어진다.

1. 기획 단계의 실제 pain point와 더 가깝다.
2. 단순 기술평가 도구보다 상위 개념의 기획 보조 시스템으로 보인다.
3. 산업공학적 의사결정 지원과도 잘 연결된다.
4. 기술혁신론 수업에서 "기술-시장-운영의 번역 문제"를 다룬다는 점이 드러난다.

## 도메인 선택에 주는 시사점
이 관점에서는 도메인이 아래 조건을 만족하면 좋다.

- 서비스 아이디어와 기반 기술 사이의 간극이 큰 분야
- 기획자는 자주 아이디어를 내지만 실제 구현 구조는 잘 모르는 분야
- 공개 자료가 충분해 기술 의존성을 설명할 수 있는 분야
- 기능 분해와 MVP 범위 설정이 중요한 분야

## 새 기준에서 유력한 도메인 예시
1. 공급망/물류 운영
- 예: "AI로 자동 발주"라고 말하기 쉽지만 실제로는 수요예측, 재고정책, 시스템 연동, 운영 규칙이 필요함

2. 제조/스마트팩토리
- 예: "비전으로 불량 검출"이라고 말하기 쉽지만 실제로는 데이터 수집, 라벨링, 현장 조명 조건, 엣지 배포 등이 필요함

3. 서비스 운영 자동화
- 예: "병원 대기 최적화"라고 말하기 쉽지만 실제로는 수요예측, 자원배치, 실시간 모니터링, 정책 제약이 필요함

## 결론
우리 프로젝트는 "기술 난이도 평가 에이전트"보다 아래처럼 설명하는 것이 더 적절하다.

"기획자의 아이디어를 구현 관점으로 번역해, 필요한 기반 기술과 기술 의존성을 구조화하고, 현실적인 MVP 범위를 제안하는 기술기획 어시스턴트 에이전트"
