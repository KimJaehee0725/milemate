# LLM-as-a-Judge 평가 체크리스트: 단계형 방법론 유효성 검증

## 1. 평가 설계

### 비교 대상

| 조건 | 정의 |
|---|---|
| **Baseline (A)** | 동일한 initial_input(비개발 기획자의 첫 요청)을 주고, 중간 단계 없이 한 번에 최종 기획서를 작성 |
| **Proposed (B)** | 동일한 initial_input으로 4단계(문제정의 → MVP범위 → 리스크검증 → 최종기획서)를 거쳐 최종 기획서를 작성 |

- 비교 단위: **동일 시나리오 × 동일 initial_input**의 최종 보고서 쌍. 8개 시나리오 전부 사용.
- 두 조건 모두 동일 생성 모델·동일 토큰 예산 상한을 사용한다. 단계를 거친 쪽이 누적 컨텍스트로 길어지는 것 자체는 허용하되, 판정에서 길이 보정(아래 §2)을 적용한다.

### 핵심 공정성 원칙

체크리스트 항목은 **"좋은 기획서가 갖춰야 할 내용 속성"** 을 측정해야 하며, 제안 방법론의 산출 형식(섹션 구조, 용어)을 측정해서는 안 된다. 형식을 기준으로 삼으면 평가가 순환 논증이 된다. 따라서:

- 판정자는 섹션 제목·문서 구조가 아니라 **해당 내용이 문서 어디에든 존재하고 구체적인지**를 본다.
- 특정 템플릿 용어(예: "developer_handoff", "decision_agenda")의 등장 여부는 점수와 무관하다.

## 2. 편향 통제 프로토콜

| 편향 | 통제 방법 |
|---|---|
| 위치 편향 | 모든 쌍을 (A,B)와 (B,A) 두 순서로 2회 판정. **양방향 일치 시에만 승패 인정**, 불일치 시 tie 처리 |
| 길이/장황함 편향 | 판정 프롬프트에 "길이는 품질이 아니다. 같은 내용이면 짧은 쪽이 우수하다" 명시 + 감점 항목 P1 운영 |
| 형식 편향 | 두 문서를 동일한 plain text로 변환(마크다운 표/헤더 수준 통일) 후 제시 |
| 자기 선호 편향 | 판정 모델은 생성 모델과 다른 모델(또는 최소한 다른 temperature·프롬프트) 사용 권장 |
| 조건 누출 | 문서에서 "1단계에서", "검증 단계 결과" 등 과정 언급을 제거하거나 마스킹한 뒤 판정. 판정자에게 조건(A/B)을 알리지 않음 |
| 판정 분산 | 시나리오당 생성 seed ≥ 3쌍, 판정도 쌍당 2회(순서 스왑) — 집계는 §6 |

## 3. 채점 규칙 (공통)

모든 평가 항목은 **원자 단위**로 정의되며, 판정자는 항목마다 아래 4개 중 하나를 부여한다.

| 판정 | 점수 | 부여 조건 |
|---|---|---|
| `pass` | 1.0 | 항목의 Pass 기준을 전부 충족 |
| `partial` | 0.5 | 항목의 Partial 기준에 해당 |
| `fail` | 0.0 | 항목의 Fail 기준에 해당 |
| `na` | 제외 | 항목에 N/A 조건이 명시돼 있고 그 조건에 해당할 때만 허용 |

- **모든 pass/partial 판정에는 문서 원문 인용이 필수다.** 인용할 문장을 찾지 못하면 fail이다. (fail은 인용 불필요 — 부재의 증명은 요구하지 않는다)
- 항목 기준에 "n개 이상" 같은 수치가 있으면 그 수치로 기계적으로 판정한다. 판정자의 재량 해석을 금지한다.
- **차원 점수는 산식으로 계산한다** (판정자가 직접 매기지 않는다):

```
차원 점수 = 1 + 4 × (획득 점수 합 / 유효 항목 수)    # na 항목은 분모에서 제외, 소수 둘째 자리 반올림
```

## 4. 평가 항목 (8개 차원 × 3~5항목 = 총 32항목)

### D1. 문제 정의 품질

**D1-1. 고통 장면의 구체성**
문제 상황이 ① 누가(역할/주체) ② 언제/어떤 상황에서 ③ 무엇이 막히는지의 3요소로 서술되는가.
- Pass: 3요소 모두 특정됨 (예: "피크타임에 운영자가 두 화면을 번갈아 보며 수동 조정")
- Partial: 2요소만 특정됨
- Fail: 1요소 이하, 또는 "고객 경험이 나쁘다" 수준의 추상 서술만 존재

**D1-2. 비즈니스 임팩트의 수치화**
문제가 방치될 때의 손실이 단위를 가진 수치(원, 건, %, 분, 일)로 표현되는가.
- Pass: 단위 있는 수치로 표현되고, 그 수치의 출처가 입력 또는 명시된 가정에서 식별 가능
- Partial: 손실의 방향성(증가/감소/누적)만 서술되고 수치 없음
- Fail: 임팩트 서술 자체가 없음

**D1-3. 현행 우회 방식과 한계**
지금 현장이 어떻게 버티고 있는지(수작업/임시 프로세스)와 그 방식의 한계가 적혀 있는가.
- Pass: 현행 방식 + 그 방식이 왜 지속 불가능한지 둘 다 서술
- Partial: 현행 방식 언급만 있고 한계 서술 없음
- Fail: 현행 방식 언급 없음

**D1-4. 성공 기준의 검증 가능성**
성공 기준 각각이 측정 대상과 판정 조건을 포함해, 제3자가 달성 여부를 yes/no로 판별할 수 있는가.
- Pass: 모든 성공 기준이 판별 가능 (예: "위험 주문을 1분 안에 식별한다")
- Partial: 성공 기준 중 일부만 판별 가능
- Fail: 성공 기준이 없거나 전부 선언적 표현 (예: "신뢰를 회복한다")

### D2. 범위 규율 (MVP scope discipline)

**D2-1. 포함 범위의 구체성**
포함 기능이 열거되고, 각 항목이 단일 사용자 흐름 또는 화면/액션 수준으로 구체적인가.
- Pass: 포함 항목이 명시적으로 열거되고 전 항목이 흐름/화면/액션 수준
- Partial: 열거는 있으나 일부 항목이 추상적 ("고객 만족 기능")
- Fail: 포함 범위 열거가 없거나 전부 추상적

**D2-2. 제외 범위의 명시**
이번에 하지 않는 것이 명시적으로 열거되는가. (적지 않은 것과 의도적으로 뺀 것의 구분)
- Pass: 제외 항목 2개 이상 명시적으로 열거됨
- Partial: 제외 항목 1개, 또는 "~는 추후 검토" 식의 간접 언급만
- Fail: 제외 범위 언급 없음

**D2-3. 자동화-사람 경계**
어떤 판단/계산이 자동이고, 어떤 최종 액션을 사람이 쥐는지 양쪽이 명시되는가.
- Pass: 자동 영역과 사람 확정 영역 둘 다 명시
- Partial: 한쪽만 명시
- Fail: 경계 서술 없음

**D2-4. 범위 결정의 근거**
"왜 이것부터 하는지"가 문제·리스크·효과와 연결되어 서술되는가.
- Pass: 우선순위 근거가 D1의 문제 또는 식별된 리스크와 명시적으로 연결됨
- Partial: 근거가 있으나 일반론 ("작게 시작하는 것이 좋다")
- Fail: 근거 서술 없음

### D3. 실행 구체성 (actionability)

**D3-1. 화면/접점 정의**
핵심 화면(또는 사용자 접점)에 대해 ① 주 사용자 ② 진입 경로 ③ 주요 액션이 정의되는가.
- Pass: 3요소 모두 정의됨
- Partial: 2요소만 정의됨
- Fail: 1요소 이하, 또는 "대시보드를 만든다" 수준의 선언만

**D3-2. 예외 상태 처리**
정상 경로 외에 ① 표시할 것이 없는 상태(empty) ② 데이터 오류/신뢰도 낮음 상태의 처리 방식이 정의되는가.
- Pass: 두 종류 모두 처리 방식 정의
- Partial: 한 종류만 정의
- Fail: 예외 상태 언급 없음

**D3-3. 운영 정책의 실행 가능성**
운영 정책이 ① 트리거(언제 발동) ② 규칙(무엇을 한다) ③ 예외 처리 ④ 담당 조직의 4요소로 적혀 있는가.
- Pass: 4요소 모두 존재
- Partial: 2~3요소 존재
- Fail: 1요소 이하, 또는 정책 서술 없음

**D3-4. 데이터 요건의 특정**
필요 데이터 각각에 ① 출처 ② 용도 ③ 최신성(freshness) 또는 품질 조건이 적혀 있는가.
- Pass: 열거된 모든 데이터에 3요소 충족
- Partial: 출처·용도는 있으나 최신성/품질 조건이 없음, 또는 일부 데이터만 충족
- Fail: "데이터가 필요하다" 수준이거나 데이터 언급 없음

**D3-5. 측정용 로그/이벤트 설계**
어떤 행동·시점에 어떤 속성을 기록하는지와, 그 로그가 어떤 분석/지표에 쓰이는지가 정의되는가.
- Pass: 기록 시점 + 기록 속성 + 사용처(분석/지표 연결) 모두 정의
- Partial: 로그 항목 나열만 있고 사용처 없음
- Fail: 로그/이벤트 설계 없음

### D4. 측정 가능성

**D4-1. KPI의 baseline과 target**
핵심 KPI 각각에 현재값(baseline)과 목표값(target)이 모두 있는가. baseline이 미측정이면 그 사실과 측정 계획을 명시해야 한다.
- Pass: 모든 핵심 KPI에 baseline + target (또는 미측정 명시 + 측정 계획)
- Partial: 일부 KPI만 충족, 또는 target만 있음
- Fail: baseline/target 구조가 전혀 없음

**D4-2. 측정 방법의 명시**
KPI별로 어떤 데이터를 어떻게 집계해 산출하는지가 적혀 있는가.
- Pass: 모든 핵심 KPI에 산출 방법 명시
- Partial: 일부 KPI만 명시
- Fail: 측정 방법 서술 없음

**D4-3. 측정 담당 주체**
KPI별 담당 주체(팀/역할)가 지정되는가.
- Pass: 모든 핵심 KPI에 담당 지정
- Partial: 일부만 지정
- Fail: 담당 지정 없음

**D4-4. 문제-KPI 정합성**
각 KPI가 D1의 문제·성공 기준 중 하나와 연결되어, KPI가 개선되면 문제가 실제로 줄어드는 논리가 성립하는가.
- Pass: 모든 핵심 KPI가 문제와 논리적으로 연결됨
- Partial: 일부 KPI의 연결이 불명확
- Fail: KPI가 문제와 무관 (예: 문제는 이탈인데 KPI는 페이지뷰)

### D5. 리스크 식별과 보류 조건

**D5-1. 시나리오 고유 실패 경로**
이 기획이 실패할 수 있는 경로가 해당 시나리오의 고유 요소(그 도메인의 데이터, 사용자, 상황)를 포함해 식별되는가.
- Pass: 시나리오 고유 명사를 포함한 실패 경로 2개 이상
- Partial: 고유 실패 경로 1개, 또는 고유·일반론 혼재
- Fail: 실패 경로 없음 또는 전부 일반론 ("데이터 품질이 중요하다")

**D5-2. 보류/중단 발동 조건**
"X 조건이면 Y 판단을 멈춘다/보류한다" 형태의 발동 조건이 명시되는가.
- Pass: 조건 → 중단/보류 행동의 형태로 1개 이상 명시 (예: "표본이 기준 미만이면 신호를 보류")
- Partial: 보류가 필요하다는 언급은 있으나 발동 조건이 없음
- Fail: 보류/중단 개념 자체가 없음

**D5-3. 과잉 자동화·과잉 개입 가드레일**
과잉 노출, 과잉 연락, 오답 응대 등이 일으킬 부작용에 대한 상한·임계 기준이 있는가.
- Pass: 정량적 또는 운영 가능한 상한/임계 기준 명시 (예: "일일 연락량 상한", "신뢰도 임계치 미만이면 자동 응답 금지")
- Partial: 부작용 우려 언급만 있고 기준 없음
- Fail: 과잉 자동화 부작용 언급 없음

**D5-4. 개인정보·민감 정보 취급**
시나리오에 개인 식별/위치/연락처 등 민감 데이터가 등장할 때, 그 식별과 취급 원칙(최소 수집, 동의, 마스킹 등)이 다뤄지는가.
- Pass: 민감 데이터 식별 + 취급 원칙 1개 이상 명시
- Partial: 민감 데이터 언급만 있고 원칙 없음
- Fail: 시나리오에 민감 데이터가 있는데 무언급
- N/A: 시나리오 입력과 범위에 민감 데이터가 등장하지 않는 경우

### D6. 내적 일관성

**D6-1. 문제→범위→지표 흐름의 단절 여부**
포함 범위의 각 기능이 문제와 연결되고, 그 효과를 잴 KPI 또는 로그가 존재하는가. (범위에 없는 기능의 KPI, 측정 경로 없는 기능 = 단절)
- Pass: 단절 0건
- Partial: 단절 1~2건 (치명적이지 않음)
- Fail: 단절 3건 이상

**D6-2. 문서 내 모순 여부**
문서 내 상호 모순(제외했다던 기능을 뒤에서 전제, 동일 지표의 수치 불일치 등)이 있는가.
- Pass: 모순 0건
- Partial: 사소한 표기 불일치 1~2건 (수치 단위, 명칭 등)
- Fail: 핵심 주장 간 모순 1건 이상 (예: "자동 발송 제외"라면서 KPI는 자동 발송률)

**D6-3. 정책-설계 충돌 여부**
제시한 가드레일·정책이 화면 설계나 범위와 충돌하지 않는가. (예: "운영자 확인 필수"인데 화면에 확인 액션이 없음)
- Pass: 충돌 없음
- Fail: 충돌 1건 이상 (이 항목은 partial 없음)

### D7. 의사결정·인계 준비도

**D7-1. 결정 안건의 완결성**
지금 결정해야 할 안건이 ① 안건 ② 선택지 2개 이상 ③ 결정 주체와 함께 제시되는가.
- Pass: 3요소를 갖춘 안건 1개 이상
- Partial: 안건과 주체는 있으나 선택지가 없거나 1개
- Fail: 결정 안건 없음

**D7-2. 미해결 질문의 추적 가능성**
열린 질문이 ① 질문 ② 답변 주체 ③ 답변 기한과 함께 남겨지는가.
- Pass: 3요소를 갖춘 질문 1개 이상
- Partial: 질문만 있고 주체 또는 기한 없음
- Fail: 미해결 질문 없음 (모든 것이 해결된 듯 서술하는 것도 fail)

**D7-3. 개발팀 인계 질문/요청 목록**
비개발 기획자가 개발팀에 그대로 들고 가서 물어볼 수 있는 구체 질문(데이터 가용성, 시스템 연결, 공수 가늠)이 있는가.
- Pass: 구체적 기술 확인 질문/요청 3개 이상 (예: "주문 상태와 위치 데이터를 시간 기준으로 조인할 수 있는가")
- Partial: 1~2개, 또는 "개발팀과 협의 필요" 수준의 추상 요청만
- Fail: 인계 질문/요청 없음

**D7-4. 결재 판단 완결성**
결재권자가 이 문서만 읽고 진행/보류를 판단할 수 있는가 — ① 무엇을 하겠다는 요약 ② 왜 해야 하는지의 근거 ③ 승인 시 무엇이 필요한지(자원/결정)의 3요소.
- Pass: 3요소 모두 존재
- Partial: 2요소 존재
- Fail: 1요소 이하 — 읽어도 다음 행동이 도출되지 않음

### D8. 입력 충실도와 절제

**D8-1. 화자의 우려·제약 반영**
initial_input에 명시된 화자의 우려/제약(예: "기사님 반발이 무섭다", "시스템 갈아엎는 건 아니다")이 각각 설계 결정(범위, 정책, 가드레일)에 대응되는가.
- Pass: 입력에서 식별되는 모든 우려/제약이 설계 요소에 대응됨
- Partial: 일부 우려만 대응됨
- Fail: 우려/제약이 무시됨
- N/A: 입력에 명시적 우려/제약이 없는 경우 (8개 시나리오 입력에는 모두 존재하므로 사실상 미사용)

**D8-2. 수치의 출처 추적 가능성**
문서의 모든 수치가 ① 입력에서 왔거나 ② 가정임이 명시되거나 ③ 측정 계획과 함께 제시되는가.
- Pass: 출처 불명 수치 0건
- Partial: 출처 불명 수치 1~2건
- Fail: 출처 불명 수치 3건 이상

**D8-3. 요청 방향 유지**
입력이 요청한 방향(신규 서비스 런칭 vs 기존 서비스 수정)이 유지되는가.
- Pass: 방향 유지
- Fail: 방향이 바뀜 (예: 화면 개선 요청에 신규 플랫폼 구축 제안) — partial 없음

**D8-4. 효과 주장의 절제**
기대 효과 주장에 조건·근거가 동반되고, 보장형 단정("반드시", "즉시 해결")이 없는가.
- Pass: 모든 효과 주장이 조건부/근거 동반
- Partial: 무조건 단정 1~2건
- Fail: 보장형 표현 3건 이상

## 5. 감점 항목 (차원 점수와 별도, 문서 총점에서 차감)

| 코드 | 항목 | 판정 기준 | 차감 |
|---|---|---|---|
| P1 | 장황함 | 실질 내용 추가 없이 같은 주장이 3회 이상 반복됨 | -0.5 |
| P2 | 보일러플레이트 | 한 섹션 전체에 시나리오 고유 명사(그 도메인의 주체·데이터·상황)가 전혀 없음 — 어느 도메인에 넣어도 성립 | 섹션당 -0.5, 최대 -1.5 |
| P3 | 근거 없는 정밀 수치 | 소수점/구체 단위의 수치인데 입력·가정·측정 계획 어디에도 근거 없음 (D8-2와 별개로, 특히 효과 예측치에 적용) | 건당 -0.5, 최대 -1.5 |
| P4 | 입력 제약 위반 | 입력이 명시적으로 배제한 것을 제안 (예: "전면 교체는 아니다" → 전면 교체 제안) | 건당 -1.0 |

감점도 **반드시 원문 인용**과 함께 부여한다.

## 6. 판정자 출력 형식

판정자는 문서 쌍마다 아래 JSON을 출력한다. 32개 항목 전부에 대해 두 문서를 각각 판정한다.

```json
{
  "scenario_id": "return_pickup_flow",
  "pair_id": "seed1_order_AB",
  "items": [
    {
      "item_id": "D1-1",
      "doc1": {"grade": "pass", "evidence": "원문 인용..."},
      "doc2": {"grade": "partial", "evidence": "원문 인용...", "missing": "무엇이 부족한지 한 문장"}
    }
  ],
  "penalties": {
    "doc1": [{"code": "P3", "evidence": "원문 인용"}],
    "doc2": []
  },
  "dimension_scores": {
    "doc1": {"D1": 4.0, "D2": 3.5, "D3": 0, "D4": 0, "D5": 0, "D6": 0, "D7": 0, "D8": 0},
    "doc2": {"D1": 2.5, "D2": 5.0, "D3": 0, "D4": 0, "D5": 0, "D6": 0, "D7": 0, "D8": 0}
  },
  "doc1_total": 0.0,
  "doc2_total": 0.0,
  "overall_verdict": "doc1 | doc2 | tie",
  "confidence": "high | medium | low"
}
```

- `grade`는 `pass | partial | fail | na`만 허용. `na`는 항목 정의에 N/A 조건이 있는 D5-4, D8-1에서만 허용된다.
- `dimension_scores`와 총점은 §3 산식으로 **판정 후 외부 스크립트에서 재계산해 검증**한다 (판정자 산수 오류 방지).
- `doc1`/`doc2`는 제시 순서 기준 익명 라벨이다. 조건(A/B) 매핑은 판정 후 외부에서 복원한다.
- 차원별 우열은 항목 점수에서 자동 도출되므로, 판정자의 핵심 임무는 **항목 단위 grade와 인용**이다.

## 7. 집계 방법

1. **차원별 승률**: 순서 스왑 2회 판정이 일치한 쌍만 win/loss로 집계, 불일치는 tie. 차원별로 B(제안)의 승률과 평균 점수 차를 보고한다.
2. **항목별 분석**: 32개 항목 각각의 pass율 차이를 보고한다 — 방법론이 **정확히 어떤 속성**(예: D2-2 제외 범위, D5-2 보류 조건)을 개선하는지가 핵심 결과물이다.
3. **총점**: 기본 가중치는 8개 차원 균등(각 12.5%). 연구 질문에 따라 D5·D7(방법론이 기여한다고 주장하는 차원)에 가중을 둔 민감도 분석을 별도 보고한다 — 단, 주 결과는 균등 가중으로 제시해 순환 논증을 피한다.
4. **통계**: 시나리오 8개 × seed 3 = 24쌍 기준, 승률에 대해 부트스트랩 95% CI 또는 sign test를 보고한다. n이 작으므로 효과 크기(평균 점수 차)를 함께 제시한다.
5. **정성 분석**: confidence가 low인 판정, 항목 grade가 스왑 판정 간 2단계 이상 차이 나는 항목(pass↔fail)은 사람이 표본 검토한다(판정 신뢰도 검증, 최소 20%).
