# LLM-as-a-Judge 평가 체크리스트: 단계형 방법론 유효성 검증

## 1. 평가 설계

### 비교 대상

| 조건 | 정의 |
|---|---|
| **Baseline (A)** | 동일한 initial_input(비개발 기획자의 첫 요청)을 주고, 중간 단계 없이 한 번에 최종 기획서를 작성 |
| **Proposed (B)** | 동일한 initial_input으로 4단계(문제정의 → MVP범위 → 리스크검증 → 최종기획서)를 거쳐 최종 기획서를 작성 |

- 비교 단위: **동일 시나리오 × 동일 initial_input**의 최종 보고서 쌍. 8개 시나리오 전부 사용.
- 두 조건 모두 동일 생성 모델·동일 토큰 예산 상한을 사용한다. 단계를 거친 쪽이 누적 컨텍스트로 길어지는 것 자체는 허용하되, 판정에서 길이 보정(아래 §2)을 적용한다.

### 핵심 공정성 원칙

체크리스트 항목은 **"좋은 기획서가 갖춰야 할 내용 속성"** 을 측정해야 하며, 제안 방법론의 산출 형식(섹션 구조, 용어)을 측정해서는 안 된다. 형식을 기준으로 삼으면 평가가 순환 논증이 된다. 따라서:

- 판정자는 섹션 제목·문서 구조가 아니라 **해당 내용이 문서 어디에든 존재하고 구체적인지**를 본다.
- 특정 템플릿 용어(예: "developer_handoff", "decision_agenda")의 등장 여부는 점수와 무관하다.

## 2. 편향 통제 프로토콜

| 편향 | 통제 방법 |
|---|---|
| 위치 편향 | 모든 쌍을 (A,B)와 (B,A) 두 순서로 2회 판정. **양방향 일치 시에만 승패 인정**, 불일치 시 tie 처리 |
| 길이/장황함 편향 | 판정 프롬프트에 "길이는 품질이 아니다. 같은 내용이면 짧은 쪽이 우수하다" 명시 + 감점 항목 P1 운영 |
| 형식 편향 | 두 문서를 동일한 plain text로 변환(마크다운 표/헤더 수준 통일) 후 제시 |
| 자기 선호 편향 | 판정 모델은 생성 모델과 다른 모델(또는 최소한 다른 temperature·프롬프트) 사용 권장 |
| 조건 누출 | 문서에서 "1단계에서", "검증 단계 결과" 등 과정 언급을 제거하거나 마스킹한 뒤 판정. 판정자에게 조건(A/B)을 알리지 않음 |
| 판정 분산 | 시나리오당 생성 seed ≥ 3쌍, 판정도 쌍당 2회(순서 스왑) — 집계는 §6 |

## 3. 평가 차원 체크리스트 (8개 차원, 차원당 1–5점)

각 차원은 세부 체크 항목으로 구성된다. 판정자는 각 세부 항목에 **pass / partial / fail**을 매기고 **반드시 문서에서 근거 문장을 인용**한 뒤, 앵커 기준으로 차원 점수(1–5)를 부여한다.

### D1. 문제 정의 품질

- [ ] 고객/현장의 고통이 구체적 장면으로 서술되는가 (누가, 언제, 무엇 때문에)
- [ ] 비즈니스 임팩트가 수치 또는 측정 가능한 형태로 연결되는가 (비용, 이탈, 건수 등)
- [ ] 현재의 우회 방식(현행 프로세스의 한계)이 명시되는가
- [ ] 성공 기준이 검증 가능한 문장으로 적혀 있는가 (달성 여부를 제3자가 판별 가능)

> 앵커 — **5**: 네 항목 모두 구체적이고 초기 입력의 문제의식과 정합. **3**: 문제는 서술되나 임팩트·성공 기준 중 하나가 모호하거나 일반론. **1**: 문제가 추상적 슬로건 수준("고객 경험 개선")이거나 입력과 무관.

### D2. 범위 규율 (MVP scope discipline)

- [ ] 포함 범위(in-scope)가 첫 릴리스로 실현 가능한 크기로 한정되는가
- [ ] **제외 범위(out-of-scope)가 명시적으로 적혀 있는가** (적지 않은 것과 뺀 것을 구분)
- [ ] 자동화와 사람 판단의 경계가 명시되는가 (무엇이 자동이고, 최종 액션은 누가 쥐는가)
- [ ] 범위 결정에 대한 근거(왜 이것부터인지)가 있는가

> 앵커 — **5**: in/out 모두 명시 + 자동/사람 경계 + 근거. **3**: in-scope는 구체적이나 out-of-scope나 경계가 암묵적. **1**: 기능 나열만 있고 경계·제외가 없음(전부 다 하겠다는 식).

### D3. 실행 구체성 (actionability)

- [ ] 핵심 화면/사용자 접점이 사용자·진입점·주요 액션 수준으로 구체화되는가
- [ ] 정상 경로 외 **빈 상태(empty)·오류/저신뢰 상태의 처리**가 정의되는가
- [ ] 운영 정책이 트리거 → 규칙 → 예외 처리 → 담당 조직 구조로 실행 가능하게 적혀 있는가
- [ ] 필요한 데이터가 출처·최신성(freshness)·품질 조건 수준으로 특정되는가
- [ ] 효과 측정에 필요한 로그/이벤트가 어떤 시점에 무엇을 남기는지 정의되는가

> 앵커 — **5**: 개발팀이 추가 회의 1회 내로 견적 가능한 수준. **3**: 화면·데이터는 있으나 예외 상태나 로그 설계가 빠짐. **1**: "대시보드를 만든다" 수준의 선언만 존재.

### D4. 측정 가능성

- [ ] KPI마다 baseline(현재값)과 target(목표값)이 모두 있는가
- [ ] 측정 방법(어떤 데이터를 어떻게 집계하는지)이 적혀 있는가
- [ ] KPI 담당 주체가 지정되는가
- [ ] KPI가 D1의 문제·성공 기준과 논리적으로 연결되는가 (개선 시 문제가 실제로 줄어드는 지표인가)

> 앵커 — **5**: 전 KPI에 baseline/target/측정법/오너 + 문제 정합. **3**: 목표는 있으나 baseline 또는 측정법 부재. **1**: "만족도 향상" 류의 측정 불가 지표.

### D5. 리스크 식별과 보류 조건

- [ ] 이 기획이 실패할 수 있는 구체적 경로(데이터 부족, 오탐/과잉 노출, 현장 반발 등)가 식별되는가
- [ ] 데이터 품질·표본이 부족할 때 **어떤 판단을 멈추거나 보류하는지** 기준이 있는가
- [ ] 과잉 자동화·과잉 개입(고객 피로, 감시 우려, 오답 응대 등)에 대한 가드레일이 있는가
- [ ] 개인정보·민감 정보 취급 리스크가 다뤄지는가 (해당 시나리오에 존재할 경우)
- [ ] 리스크가 일반론("데이터 품질이 중요하다")이 아니라 이 시나리오 고유의 조건으로 서술되는가

> 앵커 — **5**: 시나리오 고유 리스크 + 보류/롤백 발동 조건이 운영 가능한 수준. **3**: 리스크 나열은 있으나 보류 조건·대응이 없음. **1**: 리스크 섹션이 없거나 전부 일반론.

### D6. 내적 일관성

- [ ] 문제 → 범위 → KPI → 로그/데이터가 한 줄기로 이어지는가 (범위에 없는 기능의 KPI, KPI 없는 기능 등 단절이 없는가)
- [ ] 문서 내 상호 모순이 없는가 (예: out-of-scope로 뺀 자동화를 뒤에서 전제, 수치 불일치)
- [ ] 제안한 가드레일·정책이 범위 및 화면 설계와 충돌하지 않는가

> 앵커 — **5**: 단절·모순 없음. **3**: 사소한 단절 1–2건(치명적이지 않음). **1**: 핵심 주장 간 모순(예: 자동 발송 제외라더니 KPI는 자동 발송률).

### D7. 의사결정·인계 준비도

- [ ] 지금 결정해야 할 안건이 선택지·결정 주체와 함께 제시되는가
- [ ] 미해결 질문이 "누가, 언제까지" 답해야 하는지와 함께 남겨져 있는가
- [ ] **비개발 기획자가 개발팀에 그대로 들고 가서 물어볼 수 있는 질문/요청 목록**이 있는가 (데이터 가용성, 시스템 연결, 공수 가늠)
- [ ] 결재권자가 이 문서만 읽고 진행/보류를 판단할 수 있는가 (요약-근거-요청사항의 완결성)

> 앵커 — **5**: 결정 안건 + 기한 있는 미해결 질문 + 개발팀 질문지 모두 실사용 가능. **3**: 일부 존재하나 "추후 논의" 식으로 비어 있음. **1**: 다음 행동이 도출되지 않는 문서.

### D8. 입력 충실도와 절제

- [ ] initial_input에 담긴 화자의 제약·두려움(예: "기사님 반발이 무섭다", "자동 응대 오답이 걱정")이 실제로 다뤄지는가
- [ ] 입력에 없는 사실·수치를 지어내지 않았는가 (시나리오 입력에서 추론 가능한 범위는 허용, 출처 불명 수치는 fail)
- [ ] 입력이 요청한 방향(신규 런칭 vs 기존 수정)을 유지하는가
- [ ] 실현 근거 없는 약속(전면 자동화, 즉시 효과 보장 등)으로 과잉 확장하지 않았는가

> 앵커 — **5**: 화자의 우려가 설계 결정에 반영 + 허구 수치 없음. **3**: 입력은 따르나 우려 일부를 무시. **1**: 입력과 다른 문제를 풀거나 근거 없는 수치·약속 다수.

## 4. 감점 항목 (차원 점수와 별도, 최종점에서 차감)

| 코드 | 항목 | 차감 |
|---|---|---|
| P1 | 동일 내용의 반복·동어반복으로 분량만 늘림 (장황함) | -0.5 |
| P2 | 어느 도메인에 넣어도 성립하는 보일러플레이트 섹션 (시나리오 고유성 없음) | 건당 -0.5, 최대 -1.5 |
| P3 | 출처·근거 없는 정밀 수치 제시 (예: 임의의 "37.2% 개선") | 건당 -0.5, 최대 -1.5 |
| P4 | 입력의 명시적 제약 위반 (예: "시스템 전면 교체는 아니다"라는 입력에 전면 교체 제안) | -1.0 |

## 5. 판정자 출력 형식

판정자는 문서 쌍마다 아래 JSON을 출력한다.

```json
{
  "scenario_id": "return_pickup_flow",
  "pair_id": "seed1_order_AB",
  "per_dimension": [
    {
      "dimension": "D1",
      "doc1_score": 4,
      "doc2_score": 2,
      "doc1_evidence": "원문 인용...",
      "doc2_evidence": "원문 인용...",
      "verdict": "doc1",
      "rationale": "한두 문장"
    }
  ],
  "penalties": {
    "doc1": [{"code": "P3", "evidence": "원문 인용"}],
    "doc2": []
  },
  "doc1_total": 0.0,
  "doc2_total": 0.0,
  "overall_verdict": "doc1 | doc2 | tie",
  "confidence": "high | medium | low"
}
```

- `doc1`/`doc2`는 제시 순서 기준 익명 라벨이다. 조건(A/B) 매핑은 판정 후 외부에서 복원한다.
- verdict는 차원별로도 받는다 — 방법론이 **어느 차원에서** 이기는지가 총점 승패보다 중요한 분석 자료다.

## 6. 집계 방법

1. **차원별 승률**: 순서 스왑 2회 판정이 일치한 쌍만 win/loss로 집계, 불일치는 tie. 차원별로 B(제안)의 승률과 평균 점수 차를 보고한다.
2. **총점**: 기본 가중치는 8개 차원 균등(각 12.5%). 연구 질문에 따라 D5·D7(방법론이 기여한다고 주장하는 차원)에 가중을 둔 민감도 분석을 별도 보고한다 — 단, 주 결과는 균등 가중으로 제시해 순환 논증을 피한다.
3. **통계**: 시나리오 8개 × seed 3 = 24쌍 기준, 승률에 대해 부트스트랩 95% CI 또는 sign test를 보고한다. n이 작으므로 효과 크기(평균 점수 차)를 함께 제시한다.
4. **정성 분석**: confidence가 low인 판정과 차원별 verdict가 엇갈린 쌍은 사람이 표본 검토한다(판정 신뢰도 검증, 최소 20%).

```text
[원 요청]
{initial_input}

[문서1]
{doc1}

[문서2]
{doc2}

평가 규칙:

- 섹션 구조나 용어가 아니라, 필요한 내용이 존재하고 구체적인지를 평가합니다.

{json_schema}
```

## 8. 실행 절차 요약

1. 시나리오 8개 × seed 3으로 A/B 보고서 쌍 24개 생성 (동일 모델·동일 예산)
2. 과정 언급 마스킹 + plain text 정규화
3. 쌍마다 순서 스왑 2회 판정 (총 48회 judge 호출)
4. §6 집계 → 차원별 승률 표 + 총점 비교 + 정성 검토
5. 보고: "방법론은 전 차원에서 이긴다"가 아니라 **어느 차원에서, 얼마나** 이기는지를 주 결과로 제시
