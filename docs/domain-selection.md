# 도메인 선택 메모

이 문서는 팀 프로젝트의 초기 도메인 선정 과정을 기록한다.

## 1. 도메인 선정 기준
우리는 아래 6개 기준으로 도메인을 평가한다.

1. 문제 명확성
- 비전공 기획자가 실제로 겪는 문제를 설명하기 쉬운가
- 사용자 입력 예시를 자연스럽게 만들 수 있는가

2. 기술 난이도 설명 가능성
- 어떤 기술이 쉬운지/어려운지 구조적으로 설명할 수 있는가
- 병목과 우회 전략을 비교적 명료하게 제시할 수 있는가

3. 자료 접근성
- 논문, 산업 보고서, 제품 사례, 기술 블로그 등 공개 자료 확보가 쉬운가
- RAG나 지식베이스 구축을 위한 텍스트 자원이 충분한가

4. 시장 지식 통합 가능성
- 기술성뿐 아니라 시장성, 규제, 사용자 맥락까지 연결하기 쉬운가
- 기술혁신론 수업의 취지와 잘 맞는가

5. 데모 적합성
- 짧은 시간 안에 사용자 입력 -> 분석 결과 흐름을 보여주기 쉬운가
- 청중이 바로 이해할 수 있는 사례를 제시할 수 있는가

6. 구현 현실성
- 팀 프로젝트 범위에서 MVP를 만들 수 있는가
- 지나치게 광범위하거나 검증이 어려운 도메인은 아닌가

## 2. 후보 도메인 비교

| 도메인 | 장점 | 단점 | 수업 적합성 | 구현 난이도 | 총평 |
|---|---|---|---|---|---|
| 헬스케어 | 문제 중요성이 크고 기술/시장/규제 통합이 잘 드러남. 센서, 진단보조, 개인화 추천 등 예시가 풍부함 | 규제 이슈가 커서 실제 가능성 평가가 과도하게 어려워질 수 있음 | 매우 높음 | 중상 | 학술적으로 강하지만 범위를 잘 좁혀야 함 |
| 교육기술 | 사용자 시나리오가 익숙하고 데모가 쉬움. 개인화, 평가, 학습보조 등 사례가 다양함 | 기술적 병목이 다소 평이하게 보일 수 있어 혁신성이 약해 보일 위험 | 높음 | 중 | 팀플 MVP에 가장 무난함 |
| 제조/스마트팩토리 | 공정 최적화, 예지보전 등 기술 난이도 분석이 명확함 | 팀원과 청중이 도메인에 익숙하지 않으면 공감이 낮을 수 있음 | 높음 | 중상 | 기술 난제 설명에는 강함 |
| 모빌리티 | 자율주행, 물류, 경로 최적화 등 혁신 서사가 강함 | 시스템 범위가 커지고 실시간성/안전성 때문에 지나치게 어려워질 수 있음 | 높음 | 상 | 흥미롭지만 MVP 범위 통제가 어려움 |
| 뷰티/커머스 AI | 시장성과 개인화 추천/가상피팅 등 서비스 시나리오가 직관적임 | 기술혁신론 관점에서 다소 얕아 보일 수 있음 | 중상 | 중 | 데모는 좋지만 학술적 무게는 약할 수 있음 |
| 콘텐츠 생성 AI | 팀이 이해하기 쉽고 자료도 풍부함. 프롬프트/모델/API 선택 논의가 쉬움 | 너무 흔한 주제로 보여 차별성이 떨어질 수 있음 | 중 | 중하 | 구현은 쉬우나 수업에서 진부해 보일 수 있음 |

## 3. 1차 추천
현재 기준으로는 다음 3개가 유력하다.

1. 교육기술
- 이유: 기획자 입력 시나리오가 자연스럽고 MVP 구현이 쉽다.
- 장점: 자료 수집, 데모, 설명 가능성의 균형이 좋다.

2. 헬스케어
- 이유: 기술혁신론 맥락에서 가장 설득력이 크다.
- 주의: 규제와 임상 검증 이슈 때문에 너무 넓게 잡으면 위험하다.

3. 제조/스마트팩토리
- 이유: 기술 난이도와 병목 설명이 명확하다.
- 주의: 사용자 친숙도가 떨어질 수 있어 사례 선정이 중요하다.

## 4. 현재 추천 결론
수업 팀플의 안정성과 설명력을 함께 고려하면 1순위는 교육기술, 2순위는 헬스케어로 본다.

### 교육기술을 1순위로 두는 이유
- 기술 비전공 기획자의 아이디어 예시를 만들기 쉽다.
- 개인화 추천, 자동 피드백, 학습 분석, 튜터링 등 기능 분해가 자연스럽다.
- 공개 자료와 서비스 사례가 많다.
- 데모에서 입력과 출력의 품질을 바로 확인하기 쉽다.
- 규제 부담이 헬스케어보다 낮아 MVP 설계가 현실적이다.

### 헬스케어를 2순위로 두는 이유
- 기술혁신론 수업에서는 가장 임팩트가 강할 수 있다.
- 다만 기술적 난이도 외에 규제와 검증 문제가 너무 커져, 에이전트가 실제 개발 난이도보다 제도 이슈 설명에 치우칠 수 있다.

## 5. 다음 단계 제안
1. 교육기술과 헬스케어 중 하나를 우선 확정한다.
2. 확정된 도메인 안에서 세부 문제 2~3개를 뽑는다.
3. 각 문제에 대해 에이전트 출력 포맷을 설계한다.
4. 이후 자료 수집 범위를 정의한다.
