# 라스트마일 서비스 구체화

## 현재 선택한 서비스 방향
도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

## 서비스 한 줄 설명
실시간 주문 상황과 기사 상태를 반영해 배송 순서와 배차 우선순위를 재추천하는 운영 의사결정 보조 서비스

## 문제 상황
라스트마일 운영에서는 아래 문제가 동시에 발생한다.
- 주문이 시간대별로 몰린다.
- 기사별 현재 위치와 적재 상태가 계속 바뀐다.
- 고객 약속 시간과 배송 우선순위가 다르다.
- 경로가 비효율적이면 비용이 증가하고 지연이 늘어난다.
- 운영자는 실시간으로 재배차 판단을 해야 하지만 정보가 흩어져 있다.

즉, 기획자는 "실시간으로 가장 효율적인 배송 순서를 추천해주면 좋겠다"고 쉽게 말할 수 있지만, 실제 구현에는 데이터 통합, 최적화, 정책 제약, 운영자 인터페이스가 함께 필요하다.

## 핵심 사용자
- 라스트마일 배송 운영 관리자
- 배차/관제 담당자
- 물류 플랫폼 운영기획자

## 기획자가 떠올릴 수 있는 아이디어 문장
"도심 당일배송 상황에서 주문이 몰리거나 지연 위험이 생길 때, 운영자가 빠르게 배차 우선순위를 조정할 수 있도록 실시간 추천을 제공하는 서비스를 만들고 싶다."

## 이 서비스를 위해 기획자가 고려해야 할 요소
### 1. 문제 정의와 목표
- 누구의 문제를 해결하는가: 운영 관리자, 기사, 고객 중 누구를 우선할 것인가
- 어떤 KPI를 개선할 것인가: 지연률, 배송 성공률, 기사 생산성, 운영비용, 고객 불만 중 무엇이 핵심인가
- 어떤 상황에서 서비스 가치가 가장 큰가: 주문 폭주, 피크타임, 특정 지역 혼잡, 기사 부족 상황 등

### 2. 사용자와 운영 맥락
- 실제 사용자는 누구이며 언제 서비스를 쓰는가
- 운영자가 추천안을 받아들이는 방식은 무엇인가
- 현장에서는 완전 자동화보다 추천형이 더 현실적인가
- 기사 앱, 관제 대시보드, CS 시스템과 어떤 관계를 가지는가

### 3. 서비스 범위와 MVP
- 처음부터 자동 배차를 할지, 추천안 제시만 할지
- 한 도시 전체를 대상으로 할지, 특정 구역만 대상으로 할지
- 전체 주문을 다룰지, 긴급 주문이나 지연 위험 주문만 다룰지
- 실시간 기능을 얼마나 자주 업데이트할지

### 4. 운영 정책과 제약
- 약속 시간 준수와 비용 절감 중 무엇을 우선할지
- 특정 고객군, 신선식품, 당일배송 등 우선순위 정책이 있는지
- 운영자가 추천안을 무시하거나 수정할 수 있어야 하는지
- 성과가 좋아 보여도 현장에서 채택되지 않을 이유는 없는지

### 5. 사업성과 도입 가능성
- 이 서비스가 실제로 줄여주는 비용 또는 개선하는 지표는 무엇인지
- 도입 대상 기업은 어떤 규모와 운영 구조를 가진 곳인지
- 기존 솔루션 대비 차별점은 무엇인지
- 현장에서 데이터와 시스템 연동이 가능한지

## 이 서비스를 위해 기술자가 고려해야 할 요소
### 1. 데이터
- 주문 데이터, 기사 상태 데이터, 위치 데이터, 시간약속 데이터가 실제로 확보 가능한가
- 데이터가 실시간으로 들어오는가, 아니면 지연과 누락이 많은가
- 배차 결과와 실제 배송 결과를 연결해 학습/평가할 수 있는가
- 지역, 시간대, 날씨, 교통 등 외생변수를 붙일 수 있는가

### 2. 문제 모델링
- 이 문제를 규칙 기반 추천으로 시작할지, 최적화 모델로 풀지, ML을 섞을지
- 목적함수는 무엇인가: 이동거리 최소화, 지연 최소화, SLA 준수율 향상 등
- 제약조건은 무엇인가: 적재량, 근무시간, 고객 시간창, 지역 제한
- 실시간 재계산 주기를 어느 수준으로 둘 것인가

### 3. 시스템 연동
- 주문 관리 시스템, 기사 앱, 관제 시스템과 어떻게 연결할 것인가
- 추천 결과를 어떤 인터페이스로 운영자에게 보여줄 것인가
- 운영자가 수동 수정한 내역을 다시 시스템에 반영할 수 있는가
- 시스템 장애 시 기존 운영 방식으로 fallback 가능한가

### 4. 성능과 평가
- 추천안이 실제 운영에 도움이 되는지 어떻게 평가할 것인가
- 오프라인 시뮬레이션과 온라인 테스트를 어떻게 구분할 것인가
- 처리 속도 요구사항은 어느 정도인가
- 추천안 설명 가능성이 필요한가

### 5. 현실적 MVP 설계
- 처음부터 최적화 모델을 정교하게 만들지 말고 규칙 기반 추천으로 시작할 수 있는가
- 특정 지역/시간대/배송 유형만 대상으로 좁힐 수 있는가
- 완전 자동 실행이 아니라 운영자 추천안 제시 수준으로 시작할 수 있는가
- 실제 데이터를 못 구하면 공개 사례나 합성 데이터로 구조를 검증할 수 있는가

## 필요한 기반 기술
- 주문/기사/차량 상태 통합 데이터 처리
- 실시간 위치 및 이벤트 수집
- 배차/라우팅 추천 로직
- ETA 계산 또는 지연 위험 추정
- 운영 제약조건 반영 로직
- 관제 대시보드 또는 추천 인터페이스
- 운영자 override 및 로그 저장 기능

## 현실적인 MVP 제안
### MVP 범위
- 특정 도심 구역 대상
- 당일배송 주문 중 지연 위험이 큰 주문 우선 대응
- 운영자용 대시보드에서 재배차 추천안 제공
- 단순 규칙 기반 또는 제한된 최적화 기반 추천

### MVP에서 제외해도 되는 것
- 전체 도시 단위 완전 자동 배차
- 모든 기사와 모든 주문의 실시간 최적화
- 고도화된 강화학습 기반 추천
- 고객 앱까지 포함한 풀 서비스 구현

## 발표용 설명 문장
라스트마일 배송 운영에서는 주문 변동과 자원 제약 때문에 운영자가 실시간으로 배차 우선순위를 재조정해야 한다. 우리는 이 문제를 해결하기 위해, 실시간 주문 및 기사 상태를 바탕으로 배송 순서와 배차 우선순위를 재추천하는 운영 의사결정 보조 서비스를 기획하고자 한다. 이 서비스는 기획자의 직관적 아이디어를 실제 구현 가능한 기능 구조와 기반 기술 요구사항으로 번역하고, 현실적인 MVP 범위를 제안하는 데 초점을 둔다.

## 데모 시나리오
### 데모 목표
청중이 아래 네 가지를 한 번에 이해하도록 만드는 것이 목표다.
- 기획자의 직관적 아이디어는 실제로는 여러 기반 기술과 운영 제약을 필요로 한다.
- 에이전트는 search, memory, tool calling, verifier 등을 활용해 이를 구조화할 수 있다.
- 중요한 의사결정은 에이전트가 독단적으로 확정하지 않고 사람에게 되돌린다.
- 결과적으로 기획자와 기술자 모두에게 쓸모 있는 산출물을 만든다.

### 데모 상황 설정
- 배경: 도심 당일배송 운영팀이 저녁 피크타임 지연 문제를 겪고 있다.
- 문제: 갑작스런 주문 증가로 특정 구역에 주문이 몰렸고, 일부 기사에게 업무가 편중되었다.
- 사용자: 물류 플랫폼 운영기획자
- 사용자의 목표: 지연률을 낮추면서도 운영비용이 과도하게 증가하지 않는 배차 재추천 보조 서비스를 기획하고 싶다.

### 데모 입력 예시
1차 입력
"도심 당일배송 서비스에서 저녁 6시 이후 특정 지역 주문이 몰릴 때 배차가 꼬이고 지연이 많이 발생합니다. 운영자가 실시간으로 기사와 주문 상태를 보고 우선순위를 다시 조정할 수 있도록 도와주는 서비스를 기획하고 싶습니다. 완전 자동화보다는 운영자가 추천안을 참고하는 형태면 좋겠고, 핵심 목표는 지연률 감소입니다. 어떤 기능이 필요하고 어떤 기술부터 준비해야 할까요?"

### stage 기반 데모 구조
이 데모는 1회성 답변이 아니라, stage-gated 상호작용으로 설계한다.
각 stage는 아래 요소를 가진다.
- 목적
- 산출물
- 사용자 승인 조건
- 다음 stage 진입 조건
- 필요시 되돌아갈 이전 stage

#### stage 1. 문제 정의 및 목표 정렬
에이전트는 먼저 아래를 수행한다.
- intent parsing
- short-term memory 저장
- retrieval + web search
- 문제 재정의 초안 생성
- KPI 후보와 서비스 범위 후보 제안

이 stage에서는 일부를 확정하지 않고 사용자에게 되돌린다.
에이전트가 사용자에게 묻는 핵심 확인 질문 예시는 다음과 같다.
- KPI 우선순위는 지연률 감소가 1순위인가, 비용 증가 억제가 1순위인가
- 추천 대상을 모든 주문으로 볼지, 지연 위험 주문으로 한정할지
- 운영자용 대시보드 중심으로 볼지, 기사 앱 개입까지 포함할지

stage 1 산출물
- 문제 재정의 초안
- KPI 우선순위 후보
- 서비스 범위 초안
- human-in-the-loop 체크포인트 목록

stage 1 완료 조건
- 사용자가 핵심 목표와 우선 KPI를 승인함
- 추천 대상 범위에 대한 1차 합의가 생김

다음 stage 진입 조건
- stage 1 decision log가 승인 상태가 됨

#### stage 2. 서비스 구조 및 MVP 범위 설계
사용자가 예를 들어 다음처럼 응답한다고 가정한다.
"비용도 중요하지만 우선은 지연률 감소가 더 중요합니다. 전체 주문보다는 지연 위험 주문만 우선 다루고 싶고, 기사 앱까지는 범위가 커서 운영자 대시보드만 먼저 하고 싶습니다."

에이전트는 이 피드백을 short-term memory에 저장한다.
그리고 아래를 수정한다.
- KPI 우선순위 업데이트
- 대상 주문 범위 축소
- UI 범위를 운영자 대시보드로 한정
- 기사 앱 연동은 후속 단계로 이동
- 기능 구조와 MVP 범위를 재작성

이 stage에서 에이전트는 수정 전/후 차이도 보여준다.
예:
- 이전안: 전체 주문 대상 재배차 추천
- 수정안: 지연 위험 주문 대상 재배차 추천

stage 2 산출물
- 기능 구조 초안
- MVP 포함/제외 항목
- 변경 전/후 비교
- 미확정 항목 목록

stage 2 완료 조건
- 사용자가 MVP 범위와 자동화 수준을 승인함

다음 stage 진입 조건
- stage 2 산출물에 대해 사용자 승인 완료

되돌아갈 수 있는 이전 stage
- 새로운 KPI 충돌이 생기면 stage 1로 rollback

#### stage 3. 기술/운영 가정 검증
에이전트는 verifier / critic을 호출해 사용자 피드백까지 반영된 안을 점검한다.
여기서 검토하는 항목은 다음과 같다.
- 이 안이 여전히 너무 큰가
- 필수 데이터 가정이 현실적인가
- 운영자가 실제로 사용할 워크플로우와 맞는가
- KPI와 기능이 정렬되어 있는가
- 누락된 제약조건이 없는가

이 stage에서는 에이전트가 다시 human-in-the-loop를 넣을 수 있다.
예를 들면:
- 기사 위치 데이터가 실시간으로 없을 경우, 이벤트 로그 기반으로 시작해도 되는가
- ETA 정밀 예측 없이 rule-based 재추천만으로 MVP를 시작해도 되는가

stage 3 산출물
- verifier 점검 결과
- 현실성 높은 데이터/낮은 데이터 구분
- 구현 우선순위 재정렬안
- 리스크 및 rollback 필요 여부

stage 3 완료 조건
- 기술적 가정이 최소 수준으로 정리되고, 치명적 모순이 없다고 확인됨

다음 stage 진입 조건
- verifier 통과 + 사용자 승인

되돌아갈 수 있는 이전 stage
- 데이터 가정이 비현실적이면 stage 2로 rollback
- KPI와 기능이 정렬되지 않으면 stage 1로 rollback

#### stage 4. 최종 합의 및 역할별 출력
최종적으로 에이전트는 아래 두 가지 결과를 분리해 낸다.
- 기획자용 최종 요약
- 기술자용 실행 초안

또한 마지막에 아래를 명시한다.
- 현재까지 확정된 것
- 아직 미확정인 것
- 다음 session 또는 다음 stage에서 결정해야 할 것

stage 4 산출물
- 기획자용 최종 문서

### 데모에서 보여줄 에이전트 동작 흐름
#### 1단계. intent parsing
- 사용자의 목표를 파악한다: 지연률 감소, 완전 자동화 아님, 운영자 보조형 서비스
- 도메인을 식별한다: 라스트마일 배송 운영
- 핵심 제약을 추출한다: 실시간성, 운영자 개입 가능성, 비용 증가 억제

#### 2단계. short-term memory 저장
- 현재 대화에서 중요한 제약을 세션 메모리에 저장한다.
- 예: 목표 KPI=지연률 감소, 서비스 형태=추천형, 우선 사용자=운영자
- stage가 진행될수록 합의된 내용, 미해결 항목, 승인 상태를 함께 저장한다.

#### 3단계. retrieval + web search
- 내부 지식베이스에서 아래 자료를 검색한다.
  - last-mile dispatching
  - vehicle routing problem with time windows
  - dynamic dispatch
  - ETA and delay management
  - courier operations dashboard
- 외부 웹 검색으로 아래를 찾는다.
  - 기업 사례: Uber, DoorDash, Amazon, 쿠팡 등 라스트마일 운영 사례
  - 실무 블로그: dispatch optimization, route replanning, delivery ETA systems
  - 산업 자료: last-mile cost drivers, SLA management, dispatch operations

#### 4단계. tool calling 기반 구조화
- 검색 결과를 바탕으로 도구를 호출해 아래 작업을 수행한다.
  - 서비스 아이디어를 기능 단위로 분해
  - 필요한 데이터 목록 추출
  - 운영 KPI 후보 정리
  - 제약조건 목록 생성
  - MVP 범위와 제외 범위 구분
  - 사용자 확인이 필요한 결정 포인트 추출

#### 5단계. planner agent
- 기능 구조를 아래처럼 정리한다.
  1. 주문/기사 상태 통합 모니터링
  2. 지연 위험 주문 탐지
  3. 재배차 추천 로직
  4. 운영자 대시보드 및 override
  5. 추천 결과 로그 저장 및 사후 분석
- 기반 기술을 아래처럼 연결한다.
  - 실시간 이벤트 파이프라인
  - 배차/라우팅 로직
  - ETA 또는 지연 위험 계산
  - 제약조건 엔진
  - UI/대시보드
- 동시에 어떤 요소가 human approval이 필요한지 표시한다.
  - KPI 우선순위
  - 자동화 수준
  - 포함할 주문 범위
  - 필수 데이터 가정

#### 6단계. long-term memory 참조
- 과거에 축적한 라스트마일 패턴이나 공통 제약을 불러온다.
- 예: 시간창 제약, 기사 근무시간, 지역 편중, 추천형 MVP 선호
- 이전 대화에서 같은 사용자가 선호한 범위 축소 패턴이 있다면 재사용할 수 있다.

#### 7단계. verifier / critic
- 결과가 사용자의 요구와 맞는지 검증한다.
- 확인 항목 예시:
  - 완전 자동화로 과도하게 범위를 확장하지 않았는가
  - 지연률 감소라는 KPI가 결과에 반영되었는가
  - 실제 확보하기 어려운 데이터를 필수 전제로 두지 않았는가
  - 운영자 override 기능이 빠지지 않았는가
  - 최신 human feedback이 결과에 반영되었는가

#### 8단계. final report generation
- 결과를 기획자용 요약과 기술자용 요약으로 나누어 출력한다.
- 추가로 decision log를 출력한다.
  - 확정된 결정
  - 보류된 결정
  - 사용자 확인이 필요한 다음 질문

### 데모 출력 예시 구조
#### A. stage 1 출력
- 문제 재정의 초안
- KPI 우선순위 후보
- 서비스 범위 초안
- 사용자 확인 질문 3개
- stage 1 decision log

#### B. stage 2 출력
- 사용자 피드백 반영 요약
- 변경 전/후 비교
- 수정된 MVP 범위
- 추가 확인이 필요한 리스크 1~2개
- stage 2 승인 요청

#### C. stage 3 출력
- verifier 점검 결과
- 현실성 높은 데이터/낮은 데이터 구분
- 구현 우선순위 재정렬
- 남아 있는 가정 표시
- rollback 필요 여부

#### D. stage 4 출력
기획자용 최종 요약
- 문제 재정의: 피크타임 지연 완화를 위한 운영자 보조형 재배차 서비스
- 핵심 사용자: 관제 담당자, 운영 관리자
- 목표 KPI: 지연률 감소, SLA 준수율 향상, 기사 업무 편중 완화
- MVP 범위: 특정 구역, 특정 시간대, 지연 위험 주문 대상 추천안 제시 중심
- 기대 효과: 운영 판단 속도 향상, 지연 대응 일관성 향상

기술자용 최종 요약
- 필수 데이터: 주문 상태, 기사 위치, 시간창, 구역 정보, 과거 지연 로그
- 핵심 기술: 실시간 데이터 수집, 간단한 배차 추천 로직, ETA/지연 위험 추정, 대시보드
- 제약조건: 시간창, 기사 근무시간, 적재량, 지역 제한
- 구현 우선순위: 규칙 기반 추천 -> 제한적 최적화 -> 고도화
- 검증 방법: 과거 시뮬레이션, 운영자 피드백, 지연률 비교

decision log
- 확정: 추천형 서비스, 운영자 대시보드 중심, 지연 위험 주문 우선
- 보류: 기사 앱 연동 여부, ETA 고도화 수준
- 다음 질문: 실시간 기사 위치 데이터 접근 가능 여부

### human-in-the-loop 설계 포인트
- 에이전트는 모든 불확실성을 스스로 메우지 않는다.
- high-impact 결정은 질문으로 되돌린다.
- 사용자 응답을 구조화된 memory 슬롯에 저장한다.
- 각 stage는 사용자 승인 전까지 잠정 상태로 남는다.
- 이후 추천안은 항상 최신 사용자 피드백을 우선한다.
- verifier는 "사용자 피드백 미반영"을 오류로 간주한다.
- 미래 stage에서 새 리스크가 나오면 이전 stage 재검토를 허용한다.

### 데모에서 강조할 현대 에이전트 요소
- stage-gated planning: 각 stage 완료 후 다음 단계로 진행
- human-in-the-loop: KPI, 범위, 자동화 수준은 사용자 승인 필요
- rollbackable workflow: 미래 stage에서 새 제약이 생기면 이전 stage로 복귀 가능
- search: 도메인 문서와 사례 검색
- web search: 최신 산업 사례와 구현 관행 탐색
- short-term memory: 현재 대화의 KPI와 제약을 유지
- long-term memory: 축적된 도메인 지식 재사용
- tool calling: 구조화, 계산, 체크리스트 생성, 보고서 작성 자동화
- planner: 기능/기술 분해 및 단계화
- verifier: 범위 과대설정, 현실성 오류, 사용자 피드백 누락 점검
- role-based output generation: 기획자용과 기술자용 결과 분리
- optional multi-agent setup: 조사 에이전트, 설계 에이전트, 검증 에이전트 분리 가능

### 발표에서 보이는 그림
1. 사용자가 자연어로 서비스 아이디어를 입력한다.
2. 에이전트가 관련 자료를 검색하고 stage 1 초안을 만든다.
3. 사용자가 stage 1을 승인해야 stage 2로 진행한다.
4. 에이전트가 MVP 구조를 제안하고, verifier가 stage 3에서 현실성을 점검한다.
5. 새 제약이 발견되면 stage 2 또는 stage 1로 rollback한다.
6. 최종적으로 기획자용/기술자용 결과와 decision log를 보여준다.

이 흐름을 보여주면, 단순 LLM 채팅이 아니라 stage-gated, human-in-the-loop 기반의 현대적 에이전트 시스템이라는 점이 분명해진다.
