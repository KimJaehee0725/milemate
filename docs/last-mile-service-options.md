# 라스트마일 서비스 구체화

## 현재 선택한 서비스 방향
도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

## 서비스 한 줄 설명
실시간 주문 상황과 기사 상태를 반영해 배송 순서와 배차 우선순위를 재추천하는 운영 의사결정 보조 서비스

## 문제 상황
라스트마일 운영에서는 아래 문제가 동시에 발생한다.
- 주문이 시간대별로 몰린다.
- 기사별 현재 위치와 적재 상태가 계속 바뀐다.
- 고객 약속 시간과 배송 우선순위가 다르다.
- 경로가 비효율적이면 비용이 증가하고 지연이 늘어난다.
- 운영자는 실시간으로 재배차 판단을 해야 하지만 정보가 흩어져 있다.

즉, 기획자는 "실시간으로 가장 효율적인 배송 순서를 추천해주면 좋겠다"고 쉽게 말할 수 있지만, 실제 구현에는 데이터 통합, 최적화, 정책 제약, 운영자 인터페이스가 함께 필요하다.

## 핵심 사용자
- 라스트마일 배송 운영 관리자
- 배차/관제 담당자
- 물류 플랫폼 운영기획자

## 기획자가 떠올릴 수 있는 아이디어 문장
"도심 당일배송 상황에서 주문이 몰리거나 지연 위험이 생길 때, 운영자가 빠르게 배차 우선순위를 조정할 수 있도록 실시간 추천을 제공하는 서비스를 만들고 싶다."

## 이 서비스를 위해 기획자가 고려해야 할 요소
### 1. 문제 정의와 목표
- 누구의 문제를 해결하는가: 운영 관리자, 기사, 고객 중 누구를 우선할 것인가
- 어떤 KPI를 개선할 것인가: 지연률, 배송 성공률, 기사 생산성, 운영비용, 고객 불만 중 무엇이 핵심인가
- 어떤 상황에서 서비스 가치가 가장 큰가: 주문 폭주, 피크타임, 특정 지역 혼잡, 기사 부족 상황 등

### 2. 사용자와 운영 맥락
- 실제 사용자는 누구이며 언제 서비스를 쓰는가
- 운영자가 추천안을 받아들이는 방식은 무엇인가
- 현장에서는 완전 자동화보다 추천형이 더 현실적인가
- 기사 앱, 관제 대시보드, CS 시스템과 어떤 관계를 가지는가

### 3. 서비스 범위와 MVP
- 처음부터 자동 배차를 할지, 추천안 제시만 할지
- 한 도시 전체를 대상으로 할지, 특정 구역만 대상으로 할지
- 전체 주문을 다룰지, 긴급 주문이나 지연 위험 주문만 다룰지
- 실시간 기능을 얼마나 자주 업데이트할지

### 4. 운영 정책과 제약
- 약속 시간 준수와 비용 절감 중 무엇을 우선할지
- 특정 고객군, 신선식품, 당일배송 등 우선순위 정책이 있는지
- 운영자가 추천안을 무시하거나 수정할 수 있어야 하는지
- 성과가 좋아 보여도 현장에서 채택되지 않을 이유는 없는지

### 5. 사업성과 도입 가능성
- 이 서비스가 실제로 줄여주는 비용 또는 개선하는 지표는 무엇인지
- 도입 대상 기업은 어떤 규모와 운영 구조를 가진 곳인지
- 기존 솔루션 대비 차별점은 무엇인지
- 현장에서 데이터와 시스템 연동이 가능한지

## 이 서비스를 위해 기술자가 고려해야 할 요소
### 1. 데이터
- 주문 데이터, 기사 상태 데이터, 위치 데이터, 시간약속 데이터가 실제로 확보 가능한가
- 데이터가 실시간으로 들어오는가, 아니면 지연과 누락이 많은가
- 배차 결과와 실제 배송 결과를 연결해 학습/평가할 수 있는가
- 지역, 시간대, 날씨, 교통 등 외생변수를 붙일 수 있는가

### 2. 문제 모델링
- 이 문제를 규칙 기반 추천으로 시작할지, 최적화 모델로 풀지, ML을 섞을지
- 목적함수는 무엇인가: 이동거리 최소화, 지연 최소화, SLA 준수율 향상 등
- 제약조건은 무엇인가: 적재량, 근무시간, 고객 시간창, 지역 제한
- 실시간 재계산 주기를 어느 수준으로 둘 것인가

### 3. 시스템 연동
- 주문 관리 시스템, 기사 앱, 관제 시스템과 어떻게 연결할 것인가
- 추천 결과를 어떤 인터페이스로 운영자에게 보여줄 것인가
- 운영자가 수동 수정한 내역을 다시 시스템에 반영할 수 있는가
- 시스템 장애 시 기존 운영 방식으로 fallback 가능한가

### 4. 성능과 평가
- 추천안이 실제 운영에 도움이 되는지 어떻게 평가할 것인가
- 오프라인 시뮬레이션과 온라인 테스트를 어떻게 구분할 것인가
- 처리 속도 요구사항은 어느 정도인가
- 추천안 설명 가능성이 필요한가

### 5. 현실적 MVP 설계
- 처음부터 최적화 모델을 정교하게 만들지 말고 규칙 기반 추천으로 시작할 수 있는가
- 특정 지역/시간대/배송 유형만 대상으로 좁힐 수 있는가
- 완전 자동 실행이 아니라 운영자 추천안 제시 수준으로 시작할 수 있는가
- 실제 데이터를 못 구하면 공개 사례나 합성 데이터로 구조를 검증할 수 있는가

## 필요한 기반 기술
- 주문/기사/차량 상태 통합 데이터 처리
- 실시간 위치 및 이벤트 수집
- 배차/라우팅 추천 로직
- ETA 계산 또는 지연 위험 추정
- 운영 제약조건 반영 로직
- 관제 대시보드 또는 추천 인터페이스
- 운영자 override 및 로그 저장 기능

## 현실적인 MVP 제안
### MVP 범위
- 특정 도심 구역 대상
- 당일배송 주문 중 지연 위험이 큰 주문 우선 대응
- 운영자용 대시보드에서 재배차 추천안 제공
- 단순 규칙 기반 또는 제한된 최적화 기반 추천

### MVP에서 제외해도 되는 것
- 전체 도시 단위 완전 자동 배차
- 모든 기사와 모든 주문의 실시간 최적화
- 고도화된 강화학습 기반 추천
- 고객 앱까지 포함한 풀 서비스 구현

## 발표용 설명 문장
라스트마일 배송 운영에서는 주문 변동과 자원 제약 때문에 운영자가 실시간으로 배차 우선순위를 재조정해야 한다. 우리는 이 문제를 해결하기 위해, 실시간 주문 및 기사 상태를 바탕으로 배송 순서와 배차 우선순위를 재추천하는 운영 의사결정 보조 서비스를 기획하고자 한다. 이 서비스는 기획자의 직관적 아이디어를 실제 구현 가능한 기능 구조와 기반 기술 요구사항으로 번역하고, 현실적인 MVP 범위를 제안하는 데 초점을 둔다.
