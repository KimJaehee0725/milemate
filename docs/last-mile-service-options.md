# 라스트마일 서비스 기획 프레임워크

## 목적
이 문서는 특정 서비스 하나만 위한 문서가 아니라, 라스트마일 관련 다양한 서비스 아이디어를 stage-based human-in-the-loop 방식으로 기획하기 위한 공통 프레임워크를 정리한다.

대표 예시로는 아래 서비스를 사용한다.
- 도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

## 이 프레임워크가 다루는 서비스 범위
이 프레임워크는 아래와 같은 라스트마일 서비스 아이디어 전반에 적용할 수 있다.
- 동적 배차 및 경로 재추천
- ETA 예측 및 지연 알림
- 실패 배송 리스크 예측 및 사전 개입
- 기사 생산성 분석 및 운영 대시보드
- 배송 슬롯 추천 및 수요 분산
- 고객 커뮤니케이션 자동화
- 배송 품질 모니터링 및 이상 징후 탐지

## 라스트마일에서 반복적으로 발생하는 공통 문제
라스트마일 운영에서는 서비스 유형이 달라도 아래 문제가 반복적으로 나타난다.
- 주문이 시간대별로 몰린다.
- 기사별 현재 위치와 적재 상태가 계속 바뀐다.
- 고객 약속 시간과 배송 우선순위가 다르다.
- 경로가 비효율적이면 비용이 증가하고 지연이 늘어난다.
- 실패 배송과 재배송이 운영 손실을 키운다.
- 운영자는 실시간으로 판단해야 하지만 정보가 흩어져 있다.
- 고객 경험, 기사 생산성, 운영비용 사이에 trade-off가 존재한다.

즉, 기획자는 직관적으로 서비스 아이디어를 낼 수 있지만, 실제 구현에는 데이터 통합, 예측, 최적화, 정책 제약, 운영자 인터페이스가 함께 필요하다.

## 라스트마일 서비스 기획에서 공통적으로 고려할 축
### 1. 문제 유형
- 배차/라우팅 문제인가
- ETA/지연 예측 문제인가
- 실패 배송/재배송 문제인가
- 고객 커뮤니케이션 문제인가
- 운영 가시화/대시보드 문제인가

### 2. 핵심 사용자
- 라스트마일 배송 운영 관리자
- 배차/관제 담당자
- 물류 플랫폼 운영기획자
- 고객경험 담당자
- 기사 운영 매니저

### 3. 핵심 KPI
- 지연률
- SLA 준수율
- 실패 배송률
- 재배송률
- 기사 생산성
- 이동거리 및 운영비용
- 고객 문의량 및 불만율

### 4. 전형적 제약조건
- 시간창
- 기사 근무시간
- 적재량
- 지역 제한
- 실시간 데이터 지연
- 기존 시스템 연동 제약
- 운영자의 수동 개입 필요성

## 기획자가 고려해야 할 공통 요소
### 1. 문제 정의와 목표
- 누구의 문제를 해결하는가
- 어떤 KPI를 우선할 것인가
- 어떤 상황에서 서비스 가치가 가장 큰가

### 2. 사용자와 운영 맥락
- 실제 사용자는 누구이며 언제 서비스를 쓰는가
- 현장에서는 완전 자동화보다 추천형이 더 현실적인가
- 기존 운영 워크플로우와 어떻게 연결되는가

### 3. 서비스 범위와 MVP
- 자동 실행인지 추천안 제시인지
- 도시 전체인지 특정 지역인지
- 전체 주문인지 특정 위험군 주문인지
- 어느 정도의 실시간성을 요구하는지

### 4. 운영 정책과 제약
- 어떤 KPI를 우선할지
- 어떤 예외 규칙을 반영해야 하는지
- 운영자가 수정하거나 무시할 수 있어야 하는지

### 5. 사업성과 도입 가능성
- 이 서비스가 실제로 줄여주는 비용 또는 개선하는 지표는 무엇인지
- 도입 대상 기업은 어떤 운영 구조를 가진 곳인지
- 기존 솔루션 대비 차별점은 무엇인지

## 기술자가 고려해야 할 공통 요소
### 1. 데이터
- 주문, 기사, 위치, 시간창, 배송 결과 데이터를 실제로 확보 가능한가
- 데이터가 실시간으로 들어오는가
- 과거 운영 로그를 학습/평가에 쓸 수 있는가
- 외생변수를 붙일 수 있는가

### 2. 문제 모델링
- 규칙 기반으로 시작할지, 최적화 모델로 풀지, ML을 섞을지
- 목적함수는 무엇인가
- 제약조건은 무엇인가
- 재계산 주기와 응답 속도 요구사항은 무엇인가

### 3. 시스템 연동
- 주문 관리 시스템, 기사 앱, 관제 시스템과 어떻게 연결할 것인가
- 결과를 어떤 인터페이스로 보여줄 것인가
- 운영자 수정 내역을 다시 학습/로그에 반영할 수 있는가
- 장애 시 fallback이 가능한가

### 4. 성능과 평가
- 추천안이 실제 운영에 도움이 되는지 어떻게 평가할 것인가
- 오프라인 시뮬레이션과 온라인 테스트를 어떻게 구분할 것인가
- 설명 가능성과 신뢰 확보가 필요한가

### 5. 현실적 MVP 설계
- 처음부터 정교한 최적화나 고성능 ML을 쓰지 않아도 되는가
- 특정 지역/시간대/배송 유형으로 좁힐 수 있는가
- 공개 사례나 합성 데이터로 구조를 먼저 검증할 수 있는가

## 라스트마일 서비스에 공통적으로 필요한 기반 기술 블록
- 주문/기사/차량 상태 통합 데이터 처리
- 실시간 위치 및 이벤트 수집
- ETA 계산 또는 지연 위험 추정
- 배차/라우팅 추천 로직
- 실패 배송 리스크 추정 로직
- 운영 제약조건 반영 엔진
- 관제 대시보드 또는 추천 인터페이스
- 운영자 override 및 로그 저장 기능
- 분석 리포트 및 성과 비교 기능

## 공통 MVP 원칙
### MVP에 우선 포함할 것
- 핵심 운영 상황 가시화
- 우선순위 대상 식별
- 추천안 또는 경고안 제시
- 운영자 개입 가능한 인터페이스
- 로그 저장과 사후 검토 기능

### 초기에 제외해도 되는 것
- 전체 도시 단위 완전 자동 의사결정
- 모든 기사와 모든 주문에 대한 실시간 최적화
- 고도화된 강화학습 기반 추천
- 고객 앱까지 포함한 풀 서비스 구현

## stage 기반 라스트마일 서비스 기획 워크플로우
각 stage는 다음 5개 요소를 가진다.
- 목적
- 산출물
- 사용자 승인 조건
- 다음 stage 진입 조건
- rollback 가능 지점

### stage 1. 문제 정의 및 목표 정렬
목적
- 사용자의 서비스 아이디어를 라스트마일 문제 유형으로 재정의한다.
- KPI와 우선순위를 정렬한다.

에이전트 작업
- intent parsing
- retrieval + web search
- 문제 재정의 초안 생성
- KPI 후보와 서비스 범위 후보 제안

산출물
- 문제 재정의 초안
- KPI 우선순위 후보
- 서비스 범위 초안
- human-in-the-loop 체크포인트 목록

사용자 승인 조건
- 핵심 목표와 우선 KPI를 승인함
- 서비스 대상 범위에 대한 1차 합의가 생김

### stage 2. 서비스 구조 및 MVP 범위 설계
목적
- 기능 구조를 만들고 MVP 범위를 자른다.

에이전트 작업
- 기능 구조 생성
- MVP 포함/제외 항목 구분
- 수정 전/후 비교 제시
- 미확정 항목 정리

산출물
- 기능 구조 초안
- MVP 포함/제외 항목
- 변경 전/후 비교
- 미확정 항목 목록

사용자 승인 조건
- MVP 범위와 자동화 수준을 승인함

rollback
- 새로운 KPI 충돌이 생기면 stage 1로 rollback

### stage 3. 기술/운영 가정 검증
목적
- 기술적 현실성과 운영 적합성을 검증한다.

에이전트 작업
- verifier / critic 실행
- 필수 데이터 현실성 점검
- 운영 워크플로우 적합성 점검
- 리스크와 가정 정리

산출물
- verifier 점검 결과
- 현실성 높은 데이터/낮은 데이터 구분
- 구현 우선순위 재정렬안
- 리스크 및 rollback 필요 여부

사용자 승인 조건
- 기술적 가정이 최소 수준으로 정리되고 치명적 모순이 없다고 확인됨
- verifier 통과 결과를 승인함

rollback
- 데이터 가정이 비현실적이면 stage 2로 rollback
- KPI와 기능이 정렬되지 않으면 stage 1로 rollback

### stage 4. 최종 합의 및 역할별 출력
목적
- 발표/프로토타입 기준안을 고정한다.

에이전트 작업
- 기획자용 최종 요약 생성
- 기술자용 실행 초안 생성
- decision log 생성
- 다음 액션 목록 정리

산출물
- 기획자용 최종 문서
- 기술자용 최종 문서
- decision log
- 다음 액션 목록

사용자 승인 조건
- 현재 버전을 발표/프로토타입 기준안으로 승인함

rollback
- 최종 출력에서 새 리스크가 발견되면 stage 2 또는 stage 3로 rollback

## stage별 에이전트 동작 흐름
1. intent parsing
- 사용자의 목표와 서비스 유형을 파악한다.

2. short-term memory
- stage가 진행될수록 합의된 내용, 미해결 항목, 승인 상태를 저장한다.

3. retrieval + web search
- 내부 지식베이스와 외부 자료를 함께 조사한다.

4. tool calling 기반 구조화
- 기능 분해, 데이터 추출, KPI 정리, 제약조건 생성, decision log 생성 등을 수행한다.

5. planner agent
- 기능 구조와 기술 블록을 설계하고 human approval이 필요한 항목을 표시한다.

6. long-term memory
- 이전 라스트마일 사례와 공통 제약 패턴을 재사용한다.

7. verifier / critic
- 범위 과대설정, 현실성 오류, 사용자 피드백 누락을 점검한다.

8. final report generation
- 기획자용과 기술자용 결과를 분리해 출력한다.

## 대표 예시 서비스
### 시나리오 1. 동적 배차 및 경로 재추천 서비스
#### 예시 서비스
도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

#### 예시 문제 재정의
저녁 피크타임에 특정 구역으로 주문이 몰릴 때, 운영자가 지연 위험 주문을 빠르게 식별하고 재배차 우선순위를 조정할 수 있도록 돕는 운영 의사결정 보조 서비스

#### 예시 입력
"저녁 피크타임에 특정 지역 주문이 몰리면 배차가 꼬이고 지연이 많이 발생합니다. 운영자가 실시간으로 기사와 주문 상태를 보고 우선순위를 다시 조정할 수 있도록 도와주는 서비스를 기획하고 싶습니다. 완전 자동화보다는 추천안을 참고하는 형태면 좋겠습니다."

#### 예시 stage 적용
stage 1
- KPI: 지연률 감소 우선
- 범위: 특정 구역, 특정 시간대
- 확인 질문: 전체 주문 대상 vs 지연 위험 주문 대상

stage 2
- MVP: 지연 위험 주문 대상 추천형 대시보드
- 포함: 주문/기사 상태 통합, 우선순위 주문 식별, 추천안 제시
- 제외: 기사 앱 직접 개입, 전체 주문 자동 배차

stage 3
- 검증: 실시간 기사 위치 데이터 확보 가능 여부, rule-based 추천으로 시작 가능한지
- rollback 조건: 기사 위치 데이터가 부정확하면 ETA 기반 보조 기능을 우선 검토

stage 4
- 출력: 기획자용 요약 + 기술자용 실행 초안 + decision log

### 시나리오 2. ETA 예측 및 지연 알림 서비스
#### 예시 서비스
라스트마일 고객경험팀을 위한 ETA 예측 및 지연 커뮤니케이션 보조 서비스

#### 예시 문제 재정의
배송 도착 예정 시간을 더 정확하게 안내하고, 지연 가능성이 높을 때 선제적으로 고객과 운영팀에 알림을 제공해 고객 불만과 문의를 줄이는 서비스

#### 예시 입력
"고객이 배송 시간을 자주 문의하고 지연되면 불만이 커집니다. 배송 도착 예정 시간을 더 정확하게 보여주고, 늦어질 것 같으면 미리 고객에게 알릴 수 있는 서비스를 기획하고 싶습니다. 고객 앱과 운영팀 대시보드에 모두 활용할 수 있으면 좋겠습니다."

#### 예시 stage 적용
stage 1
- KPI: 고객 문의량 감소, ETA 정확도 향상, SLA 준수율 향상
- 범위: 당일배송 주문 중 실시간 추적 가능한 주문
- 확인 질문: 고객 알림 우선 vs 운영팀 알림 우선, ETA 정확도 vs 구현 단순성

stage 2
- MVP: 운영팀 대시보드 중심 ETA 리스크 모니터링 + 고객 지연 알림 트리거
- 포함: ETA 추정, 지연 위험 탐지, 알림 조건 정의
- 제외: 고객별 초개인화 메시징, 고도화된 경로 재최적화

stage 3
- 검증: 위치 데이터 수집 주기와 정확도가 ETA 계산에 충분한지, 실제 알림 정책이 과도한 오탐을 만들지 않는지
- rollback 조건: ETA 정밀도가 낮으면 먼저 지연 위험 분류 서비스로 축소

stage 4
- 출력: 기획자용 요약 + 기술자용 실행 초안 + decision log

### 시나리오 3. 실패 배송 리스크 예측 및 사전 개입 서비스
#### 예시 서비스
라스트마일 운영품질팀을 위한 실패 배송 리스크 예측 및 재배송 방지 서비스

#### 예시 문제 재정의
배송 실패 가능성이 높은 주문을 미리 식별하고, 사전 연락, 시간 조정, 기사 주의 알림 같은 개입 방안을 추천해 실패 배송과 재배송 비용을 줄이는 서비스

#### 예시 입력
"배송 실패와 재배송 때문에 비용이 많이 듭니다. 실패 가능성이 높은 주문을 미리 찾아내서 고객에게 연락하거나 기사에게 주의 신호를 줄 수 있는 서비스를 기획하고 싶습니다. 처음에는 운영팀이 확인하는 형태여도 괜찮습니다."

#### 예시 stage 적용
stage 1
- KPI: 실패 배송률 감소, 재배송률 감소, 운영 손실 감소
- 범위: 실패 이력이 잦은 주문 유형 또는 특정 지역
- 확인 질문: 고객 사전 연락 중심 vs 기사 주의 알림 중심, 전체 주문 vs 고위험 주문군

stage 2
- MVP: 고위험 주문 스코어링 + 운영팀 확인 대시보드 + 개입 액션 추천
- 포함: 실패 리스크 추정, 원인 카테고리 분류, 사전 개입 추천
- 제외: 자동 고객 연락 실행, 전사 단위 통합 정책 자동화

stage 3
- 검증: 과거 실패 배송 로그가 충분히 축적되어 있는지, 실패 원인 라벨이 실제로 정의 가능한지, 추천 개입이 운영팀 워크플로우에 들어갈 수 있는지
- rollback 조건: 실패 라벨 품질이 낮으면 원인 예측 대신 규칙 기반 위험군 탐지로 축소

stage 4
- 출력: 기획자용 요약 + 기술자용 실행 초안 + decision log

## 시나리오 간 비교 포인트
- 배차 재추천은 운영 최적화와 자원배분 성격이 가장 강하다.
- ETA 예측은 고객경험과 운영 알림의 접점이 강하다.
- 실패 배송 리스크 예측은 운영품질과 비용 절감 메시지가 가장 선명하다.

세 시나리오 모두 같은 stage-based 프레임워크를 쓰지만,
- 우선 KPI,
- 필요한 데이터,
- verifier가 잡아내는 핵심 리스크,
- rollback이 자주 발생하는 지점
이 다르다는 점을 보여줄 수 있다.

## 외부 자료 확보 계획
### 1. 논문
확보 가능성: 높음

확인 근거
- arXiv에서 `last mile delivery` 검색 시 180건 이상의 결과가 확인됨.
- 배차/경로 문제는 VRP, VRPTW, dynamic dispatch, route replanning 키워드로 충분한 문헌 확보 가능
- ETA 예측은 delay prediction, travel time estimation, delivery ETA prediction 키워드로 확장 가능
- 실패 배송은 failed delivery, no-show, redelivery, delivery risk prediction 키워드로 찾는 전략이 적절

활용 방식
- 문제 정의와 기술 블록 도출
- baseline 모델 및 평가 지표 파악
- verifier 단계에서 기술 현실성 검토 근거로 사용

### 2. 기술 문서 / 구현 문서
확보 가능성: 매우 높음

확인 근거
- Google OR-Tools의 VRPTW 문서처럼 라우팅/제약조건 모델링 문서가 공개되어 있음
- 지도/ETA/알림 시스템 관련 개발 문서도 풍부함
- 엔지니어링 블로그나 공개 기술 포스트를 통해 dispatch optimization, ETA systems, operations dashboards 사례 확보 가능

활용 방식
- stage 2에서 MVP 기능 구조 설계
- stage 3에서 구현 가능성 및 시스템 연동 현실성 검토
- 프로토타입 구현 시 참고

### 3. 특허
확보 가능성: 중상

확인 근거
- Google Patents 검색 진입이 가능하고, 관련 검색과 public datasets 링크가 제공됨
- last-mile delivery, ETA, dispatch, route optimization, delivery exception 등의 키워드로 선행 아이디어 파악 가능

주의점

활용 방식

- 발표에서 산업적 relevance 보강

### 4. 법령 / 공공 규제 자료
확보 가능성: 높음

확인 근거
- `legalize-kr/legalize-kr` 저장소를 확인한 결과, 대한민국 법령을 Git 저장소 형태의 Markdown으로 관리하고 있음
- 각 법령 파일에 YAML frontmatter가 포함되어 있고, 출처가 `law.go.kr`로 연결됨
- 현재 법령 내용, 개정 이력, 시행령/시행규칙 비교, 특정 키워드 검색이 모두 Git + Markdown 기반으로 가능함
- 데이터 출처가 국가법령정보센터 OpenAPI이며, README에 원문/제개정문/신구법비교 링크 구조도 명시되어 있음

활용 방식
- 라스트마일 서비스 기획 시 개인정보, 위치정보, 전자문서/알림, 소비자 고지, 운송/물류 관련 법령 검토
- stage 3 verifier 단계에서 규제/정책 제약 확인
- 발표에서 "한국 법령 기준 검토 가능" 근거로 사용

MCP 연결 준비 방식
- mcp-hub registry에서 한국 법령 전용 MCP는 이번 확인 범위에서 직접 보이지 않았음
- 대신 아래 조합으로 사실상 연결 가능
  - GitHub MCP: `legalize-kr/legalize-kr` 저장소 검색/조회
  - File System MCP: 로컬 clone 후 법령 Markdown 검색
  - Fetch / Firecrawl MCP: law.go.kr 또는 raw GitHub 문서 fetch
- 즉 전용 법령 MCP가 없어도 Git 기반 법령 저장소를 MCP 워크플로우에 편입할 수 있음

주의점
- force-push 가능성이 README에 명시되어 있어 commit hash 고정 참조에는 주의 필요
- 규제 해석은 보조 참고용으로 쓰고, 최종 법률 자문으로 오인되지 않도록 표현 관리 필요

## 자료 확보에 대한 결론
모든 자료를 "완벽한 수준"으로 다 구할 수 있는 것은 아니다.
하지만 프로젝트 수준에서는 충분히 가능하다.

현실적으로 보면:
- 논문: 충분히 구할 수 있음
- 기술문서: 매우 잘 구할 수 있음
- 특허: 구할 수 있으나 정리 비용이 듦
- 법령/공공 규제 자료: legalize-kr를 통해 꽤 안정적으로 확보 가능
- 산업 보고서/기업 사례: 꽤 잘 구할 수 있음
- 공개 데이터셋: 일부는 충분히 구할 수 있으나, 실제 기업 내부 데이터 수준은 아님

따라서 결론은 다음과 같다.
- "필요한 자료 범주를 전부 어느 정도는 확보할 수 있다"는 답이 맞다.
- 다만 가장 취약한 부분은 실제 기업 내부 운영 데이터와 고품질 실패배송 라벨 데이터다.
- 그래서 발표와 프로토타입에서는 공개 논문 + 기술문서 + 법령 저장소 + 공개 데이터셋 + 기업 사례를 조합하는 전략이 가장 현실적이다.

## 발표에서 보이는 그림
1. 사용자가 라스트마일 서비스 아이디어를 자연어로 입력한다.
2. 에이전트가 stage 1 초안을 만들고 사용자 승인 포인트를 제시한다.
3. 사용자가 stage를 승인하거나 수정한다.
4. 에이전트가 다음 stage로 진행하거나 필요한 경우 이전 stage로 rollback한다.
5. 최종적으로 기획자용/기술자용 결과와 decision log를 보여준다.

이 흐름을 보여주면, 단일 서비스용 챗봇이 아니라 라스트마일 서비스 기획 전반에 재사용 가능한 stage-gated agent framework라는 점이 분명해진다.
