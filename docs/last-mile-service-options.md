# 라스트마일 서비스 기획 프레임워크

## 목적
이 문서는 특정 서비스 하나만 위한 문서가 아니라, 라스트마일 관련 다양한 서비스 아이디어를 stage-based human-in-the-loop 방식으로 기획하기 위한 공통 프레임워크를 정리한다.

대표 예시로는 아래 서비스를 사용한다.
- 도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

## 적용 가능한 서비스 범위
- 동적 배차 및 경로 재추천
- ETA 예측 및 지연 알림
- 실패 배송 리스크 예측 및 사전 개입
- 기사 생산성 분석 및 운영 대시보드
- 배송 슬롯 추천 및 수요 분산
- 고객 커뮤니케이션 자동화
- 배송 품질 모니터링 및 이상 징후 탐지

## 라스트마일에서 반복적으로 발생하는 공통 문제
- 주문이 시간대별로 몰린다.
- 기사별 현재 위치와 적재 상태가 계속 바뀐다.
- 고객 약속 시간과 배송 우선순위가 다르다.
- 경로가 비효율적이면 비용이 증가하고 지연이 늘어난다.
- 실패 배송과 재배송이 운영 손실을 키운다.
- 운영자는 실시간으로 판단해야 하지만 정보가 흩어져 있다.
- 고객 경험, 기사 생산성, 운영비용 사이에 trade-off가 존재한다.

즉, 기획자는 직관적으로 서비스 아이디어를 낼 수 있지만, 실제 구현에는 데이터 통합, 예측, 최적화, 정책 제약, 운영자 인터페이스가 함께 필요하다.

## 공통적으로 고려할 축
### 문제 유형
- 배차 / 라우팅 문제인가
- ETA / 지연 예측 문제인가
- 실패 배송 / 재배송 문제인가
- 고객 커뮤니케이션 문제인가
- 운영 가시화 / 대시보드 문제인가

### 핵심 사용자
- 라스트마일 배송 운영 관리자
- 배차 / 관제 담당자
- 물류 플랫폼 운영기획자
- 고객경험 담당자
- 기사 운영 매니저

### 핵심 KPI
- 지연률
- SLA 준수율
- 실패 배송률
- 재배송률
- 기사 생산성
- 이동거리 및 운영비용
- 고객 문의량 및 불만율

### 전형적 제약조건
- 시간창
- 기사 근무시간
- 적재량
- 지역 제한
- 실시간 데이터 지연
- 기존 시스템 연동 제약
- 운영자의 수동 개입 필요성

## 기획자가 고려해야 할 요소
- 누구의 문제를 해결하는가
- 어떤 KPI를 우선할 것인가
- 어떤 상황에서 서비스 가치가 가장 큰가
- 자동 실행인지 추천안 제시인지
- 도시 전체인지 특정 지역인지
- 전체 주문인지 특정 위험군 주문인지
- 기존 운영 워크플로우와 어떻게 연결되는가
- 어떤 예외 규칙을 반영해야 하는가
- 실제로 어떤 비용을 줄이거나 어떤 지표를 개선하는가

## 기술자가 고려해야 할 요소
- 주문, 기사, 위치, 시간창, 배송 결과 데이터를 실제로 확보 가능한가
- 데이터가 실시간으로 들어오는가
- 규칙 기반, 최적화 모델, ML 중 어떤 접근이 맞는가
- 목적함수와 제약조건은 무엇인가
- 재계산 주기와 응답 속도 요구사항은 무엇인가
- 기존 주문 관리 시스템, 기사 앱, 관제 시스템과 어떻게 연결할 것인가
- 오프라인 시뮬레이션과 온라인 테스트를 어떻게 구분할 것인가
- 설명 가능성과 신뢰 확보가 필요한가

## 공통 기반 기술 블록
- 주문 / 기사 / 차량 상태 통합 데이터 처리
- 실시간 위치 및 이벤트 수집
- ETA 계산 또는 지연 위험 추정
- 배차 / 라우팅 추천 로직
- 실패 배송 리스크 추정 로직
- 운영 제약조건 반영 엔진
- 관제 대시보드 또는 추천 인터페이스
- 운영자 override 및 로그 저장 기능
- 분석 리포트 및 성과 비교 기능

## 공통 MVP 원칙
### MVP에 우선 포함할 것
- 핵심 운영 상황 가시화
- 우선순위 대상 식별
- 추천안 또는 경고안 제시
- 운영자 개입 가능한 인터페이스
- 로그 저장과 사후 검토 기능

### 초기에 제외해도 되는 것
- 전체 도시 단위 완전 자동 의사결정
- 모든 기사와 모든 주문에 대한 실시간 최적화
- 고도화된 강화학습 기반 추천
- 고객 앱까지 포함한 풀 서비스 구현

## stage 기반 워크플로우
### stage 1. 문제 정의 및 목표 정렬
목적
- 사용자의 서비스 아이디어를 라스트마일 문제 유형으로 재정의한다.
- KPI와 우선순위를 정렬한다.

산출물
- 문제 재정의 초안
- KPI 우선순위 후보
- 서비스 범위 초안
- human-in-the-loop 체크포인트 목록

사용자 승인 조건
- 핵심 목표와 우선 KPI를 승인함
- 서비스 대상 범위에 대한 1차 합의가 생김

### stage 2. 서비스 구조 및 MVP 범위 설계
목적
- 기능 구조를 만들고 MVP 범위를 자른다.

산출물
- 기능 구조 초안
- MVP 포함 / 제외 항목
- 변경 전 / 후 비교
- 미확정 항목 목록

사용자 승인 조건
- MVP 범위와 자동화 수준을 승인함

rollback
- 새로운 KPI 충돌이 생기면 stage 1로 rollback

### stage 3. 기술 / 운영 가정 검증
목적
- 기술적 현실성과 운영 적합성을 검증한다.

산출물
- verifier 점검 결과
- 현실성 높은 데이터 / 낮은 데이터 구분
- 구현 우선순위 재정렬안
- 리스크 및 rollback 필요 여부

사용자 승인 조건
- 기술적 가정이 최소 수준으로 정리되고 치명적 모순이 없다고 확인됨
- verifier 통과 결과를 승인함

rollback
- 데이터 가정이 비현실적이면 stage 2로 rollback
- KPI와 기능이 정렬되지 않으면 stage 1로 rollback

### stage 4. 최종 합의 및 역할별 출력
목적
- 발표 / 프로토타입 기준안을 고정한다.

산출물
- 기획자용 최종 문서
- 기술자용 최종 문서
- decision log
- 다음 액션 목록

사용자 승인 조건
- 현재 버전을 발표 / 프로토타입 기준안으로 승인함

rollback
- 최종 출력에서 새 리스크가 발견되면 stage 2 또는 stage 3로 rollback

## 대표 시나리오
### 시나리오 1. 동적 배차 및 경로 재추천
문제 재정의
- 저녁 피크타임에 특정 구역으로 주문이 몰릴 때, 운영자가 지연 위험 주문을 빠르게 식별하고 재배차 우선순위를 조정할 수 있도록 돕는 운영 의사결정 보조 서비스

stage 포인트
- KPI: 지연률 감소 우선
- MVP: 지연 위험 주문 대상 추천형 대시보드
- 핵심 검증: 실시간 기사 위치 데이터 확보 가능 여부
- 대표 rollback: 위치 데이터가 부정확하면 ETA 기반 보조 기능으로 축소

### 시나리오 2. ETA 예측 및 지연 알림
문제 재정의
- 배송 도착 예정 시간을 더 정확하게 안내하고, 지연 가능성이 높을 때 선제적으로 고객과 운영팀에 알림을 제공해 고객 불만과 문의를 줄이는 서비스

stage 포인트
- KPI: ETA 정확도, 고객 문의량 감소, SLA 준수율 향상
- MVP: 운영팀 대시보드 중심 ETA 리스크 모니터링 + 고객 지연 알림 트리거
- 핵심 검증: 위치 데이터 수집 주기와 정확도, 알림 오탐 위험
- 대표 rollback: ETA 정밀도가 낮으면 지연 위험 분류 서비스로 축소

### 시나리오 3. 실패 배송 리스크 예측 및 사전 개입
문제 재정의
- 배송 실패 가능성이 높은 주문을 미리 식별하고, 사전 연락, 시간 조정, 기사 주의 알림 같은 개입 방안을 추천해 실패 배송과 재배송 비용을 줄이는 서비스

stage 포인트
- KPI: 실패 배송률 감소, 재배송률 감소, 운영 손실 감소
- MVP: 고위험 주문 스코어링 + 운영팀 확인 대시보드 + 개입 액션 추천
- 핵심 검증: 실패 원인 라벨 품질, 개입 추천의 워크플로우 적합성
- 대표 rollback: 라벨 품질이 낮으면 규칙 기반 위험군 탐지로 축소

## 외부 자료 확보 계획
### 논문
확보 가능성: 높음
- 라스트마일, VRP, VRPTW, dynamic dispatch, ETA prediction, failed delivery risk 키워드로 충분한 문헌 확보 가능

### 기술 문서 / 구현 문서
확보 가능성: 매우 높음
- OR-Tools 같은 라우팅 문서, ETA / 알림 / dashboard 관련 기술 문서, 엔지니어링 블로그 활용 가능

### 특허
확보 가능성: 중상
- 선행 아이디어 파악과 차별점 정리에 유용하지만, 구현 세부를 직접 제공하지는 않음

### 법령 / 공공 규제 자료
확보 가능성: 높음
- `legalize-kr/legalize-kr` 저장소와 `law.go.kr` 기반 자료 활용 가능
- 개인정보, 위치정보, 전자문서 / 알림, 소비자 고지, 운송 / 물류 관련 법령 검토 가능

### 공개 산업 데이터 / 공개 데이터셋
확보 가능성: 중상
- 공개 데이터셋은 구할 수 있지만 실제 기업 내부 데이터 수준은 아님
- 프로토타입 데모와 baseline 실험에는 충분히 활용 가능

### 산업 보고서 / 기업 사례 / 엔지니어링 블로그
확보 가능성: 높음
- Uber, DoorDash, Amazon, Instacart, 쿠팡, DHL 등 사례를 활용 가능
