# 프로젝트 개요

## 프로젝트 한 줄 설명
기획자의 서비스 아이디어를 구현 관점으로 번역해, 필요한 기반 기술과 기술 의존성을 구조화하고 현실적인 MVP 범위를 제안하는 기술기획 어시스턴트 에이전트

## 현재 주제 방향
- 메인 도메인: 공급망/물류 운영
- 세부 도메인: 라스트마일 배송 운영
- 우선 서비스 후보: 도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

## 문제의식
기획자는 기술집약적 서비스 아이디어를 떠올릴 수는 있지만, 다음 단계에서 어려움을 겪는다.
- 아이디어 실현에 어떤 기반 기술이 필요한지 모른다.
- 아이디어를 구현 가능한 기능 단위로 쪼개기 어렵다.
- 어떤 부분이 핵심 기술 의존성이 큰지 파악하기 어렵다.
- 현재 가능한 범위와 아직 어려운 범위를 구분하기 어렵다.
- 처음부터 완성형 서비스를 상상해 범위를 과도하게 크게 잡는다.

즉, 핵심 문제는 단순히 "기술이 어렵다"가 아니라 "기획 단계에서 필요한 기반 기술과 제약을 구조화하기 어렵다"는 점이다.

## 에이전트 역할
사용자가 아이디어를 입력하면 다음을 지원한다.
1. 아이디어를 구체적 문제 정의와 기능 구조로 재정의
2. 필요한 기반 기술과 기술 의존성을 식별
3. 구현 가능한 기능 단위로 분해
4. 개발 난이도와 핵심 병목을 평가
5. 현실적인 MVP 범위와 우회 전략을 제안
6. 추가 조사 키워드와 자료 방향을 안내

## 왜 이 주제가 산업공학과 잘 맞는가
- 라스트마일 배송은 배차, 라우팅, 자원배분, 서비스 수준, 운영비용 trade-off를 모두 포함한다.
- 운영 의사결정 지원이라는 산업공학의 핵심 문제의식이 잘 드러난다.
- 기술, 운영, 비용, 현장 제약을 함께 다룰 수 있다.
- 교수님과 학생들이 문제를 직관적으로 이해하기 쉽다.

## 도메인 선정 이유
공급망/물류 운영을 택한 이유는 다음과 같다.
- 산업공학과의 연결성이 매우 높다.
- 공개 자료가 풍부하다.
- 기획 아이디어와 실제 구현 구조 사이의 간극이 크다.
- 데모와 발표 시나리오를 만들기 쉽다.

라스트마일 배송 운영을 세부 도메인으로 좁힌 이유는 다음과 같다.
- 실시간 운영 의사결정이라는 특징이 뚜렷하다.
- 기반 기술 식별, 기능 분해, MVP 제안 구조를 보여주기 좋다.
- 동적 배차/경로 재추천 같은 산업공학 핵심 문제를 담을 수 있다.

## 현재 가장 유력한 서비스 정의
도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

이 서비스는 실시간 주문 및 기사 상태를 바탕으로 배송 순서와 배차 우선순위를 재추천해, 운영자가 더 빠르고 일관된 의사결정을 내릴 수 있도록 돕는 것을 목표로 한다.

## MVP 방향
처음부터 완전 자동화 시스템으로 가지 않고, 아래 기능을 우선한다.
- 실시간 주문/기사 상태 시각화
- 우선순위 주문 식별
- 배차 재추천 규칙 또는 간단한 최적화
- 운영자에게 추천안 제시

## 에이전트 시스템 구성
이 프로젝트는 단순 챗봇보다, 여러 에이전트 기능을 조합하는 현대적 에이전트 워크플로우를 지향한다.

### 핵심 기능 모듈
1. orchestrator agent
- 사용자 요청을 해석하고 전체 작업 순서를 계획한다.
- 어떤 하위 도구와 메모리를 쓸지 결정한다.
- 한 번에 답을 끝내지 않고, 어느 지점에서 사람 확인이 필요한지도 판단한다.

2. search / retrieval
- 내부 지식베이스에서 라스트마일, 배차, 라우팅, ETA, 운영 KPI 관련 문서를 검색한다.
- 필요시 웹 검색을 통해 최신 기업 사례, 블로그, 기사, 솔루션 문서를 가져온다.

3. short-term memory
- 현재 세션에서 사용자가 말한 운영 조건, 목표 KPI, 서비스 범위, 우선순위 제약을 기억한다.
- 대화 중간에 사용자가 바꾼 조건을 반영한다.
- stage 기반 상호작용에서 이전 stage의 합의사항, 미해결 쟁점, 보류된 결정을 유지한다.

4. long-term memory
- 누적된 도메인 조사 결과, 자주 등장하는 설계 패턴, 라스트마일 운영 제약 목록을 저장한다.
- 이후 다른 서비스 아이디어 분석에도 재사용한다.

5. tool calling
- 검색, 문서 조회, KPI 계산, 제약조건 체크, 템플릿 생성, 보고서 작성 같은 작업을 별도 도구 호출로 수행한다.

6. planner / decomposition
- 사용자의 서비스 아이디어를 하위 기능과 기반 기술 블록으로 분해한다.
- MVP와 이후 고도화 단계를 나눈다.
- 각 단계에서 인간 의사결정이 필요한 선택지를 명시한다.

7. verifier / critic
- 중간 산출물과 최종 결과가 사용자 목표, 운영 제약, 기술 현실성에 맞는지 검증한다.
- 과도하게 큰 범위나 빠진 전제조건이 없는지 확인한다.
- 사용자 피드백이 들어오면 기존 안과 수정안을 비교 검토한다.

8. human-in-the-loop checkpoint manager
- 에이전트가 임의로 확정하면 안 되는 항목을 사용자 승인 대상으로 분리한다.
- 예: KPI 우선순위, 자동화 수준, MVP 범위, 필요한 데이터 가정
- 현재 stage의 완료 조건과 다음 stage 진입 조건을 관리한다.
- 미래 stage에서 새로운 제약이 발견되면 어느 이전 stage로 되돌아가야 하는지도 판단한다.

9. report generator
- 기획자용 요약과 기술자용 요약을 분리해 최종 결과를 생성한다.
- 각 stage 종료 시 중간 요약본과 decision log를 생성해 사용자가 검토할 수 있게 한다.

## stage 기반 상호작용 원칙
- 에이전트는 stage 단위로 작동하며, 각 stage는 명확한 산출물과 완료 조건을 가진다.
- 사용자가 현재 stage를 승인해야 다음 stage로 진행한다.
- high-impact 의사결정은 stage 완료 전에 반드시 사람 확인을 거친다.
- 사용자 피드백은 세션 메모리에 저장하고 이후 모든 추천에 반영한다.
- 미래 stage에서 새로운 제약이나 모순이 발견되면 이전 stage로 rollback할 수 있다.
- 최종 출력 전에는 verifier가 최신 사용자 의도와 stage별 합의사항을 다시 점검한다.
