# 프로젝트 개요

## 한 줄 설명
라스트마일 관련 서비스 아이디어를 구현 관점으로 번역해, 필요한 기반 기술과 기술 의존성을 구조화하고 현실적인 MVP 범위를 제안하는 기술기획 어시스턴트 에이전트

## 현재 주제 방향
- 메인 도메인: 공급망/물류 운영
- 세부 도메인: 라스트마일 배송 운영
- 목표: 특정 서비스 하나가 아니라 라스트마일 서비스 기획 전반에 재사용 가능한 stage 기반 planning workflow 구축
- 대표 예시 서비스: 도심 당일배송 운영팀을 위한 실시간 배차 재추천 보조 서비스

## 문제의식
기획자는 기술집약적 서비스 아이디어를 떠올릴 수는 있지만, 다음 단계에서 어려움을 겪는다.
- 아이디어 실현에 어떤 기반 기술이 필요한지 모른다.
- 아이디어를 구현 가능한 기능 단위로 쪼개기 어렵다.
- 어떤 부분이 핵심 기술 의존성이 큰지 파악하기 어렵다.
- 현재 가능한 범위와 아직 어려운 범위를 구분하기 어렵다.
- 처음부터 완성형 서비스를 상상해 범위를 과도하게 크게 잡는다.

즉, 핵심 문제는 단순히 기술이 어렵다는 것이 아니라, 기획 단계에서 필요한 기반 기술과 제약을 구조화하기 어렵다는 점이다.

## 에이전트 역할
사용자가 라스트마일 관련 서비스 아이디어를 입력하면 다음을 지원한다.
1. 아이디어를 구체적 문제 정의와 기능 구조로 재정의
2. 필요한 기반 기술과 기술 의존성을 식별
3. 구현 가능한 기능 단위로 분해
4. 개발 난이도와 핵심 병목을 평가
5. 현실적인 MVP 범위와 우회 전략을 제안
6. 추가 조사 키워드와 자료 방향을 안내

## 왜 산업공학과 잘 맞는가
- 라스트마일 배송은 배차, 라우팅, 자원배분, 서비스 수준, 운영비용 trade-off를 모두 포함한다.
- 운영 의사결정 지원이라는 산업공학의 핵심 문제의식이 잘 드러난다.
- 기술, 운영, 비용, 현장 제약을 함께 다룰 수 있다.
- 교수님과 학생들이 문제를 직관적으로 이해하기 쉽다.

## 프레임워크 적용 대상 예시
- 동적 배차 및 경로 재추천 서비스
- ETA 예측 및 지연 알림 서비스
- 실패 배송 리스크 예측 및 개입 서비스
- 기사 생산성 분석 및 운영 대시보드 서비스
- 배송 슬롯 추천 및 수요 분산 서비스
- 고객 커뮤니케이션 자동화 서비스

## 공통 MVP 방향
처음부터 완전 자동화 시스템으로 가지 않고, 아래 기능을 우선한다.
- 핵심 운영 상황 가시화
- 우선순위 대상 식별
- 추천안 또는 경고안 제시
- 운영자에게 수정 가능한 인터페이스 제공

## 에이전트 시스템 개요
이 프로젝트는 단순 챗봇보다, 여러 에이전트 기능을 조합하는 stage 기반 워크플로우를 지향한다.

### 핵심 기능 모듈
1. orchestrator agent
- 사용자 요청을 해석하고 전체 작업 순서를 계획한다.
- 어느 지점에서 사람 확인이 필요한지도 판단한다.

2. retrieval / search
- 내부 지식베이스와 외부 자료를 검색한다.

3. short-term memory
- 현재 세션의 운영 조건, 목표 KPI, 서비스 범위, 우선순위 제약을 유지한다.
- stage별 합의사항과 미해결 쟁점을 기억한다.

4. long-term memory
- 누적된 도메인 조사 결과, 자주 등장하는 설계 패턴, 공통 제약을 저장한다.

5. tool calling
- 검색, 문서 조회, KPI 계산, 제약조건 체크, 템플릿 생성, 보고서 작성 같은 작업을 호출 기반으로 수행한다.

6. planner / decomposition
- 서비스 아이디어를 기능과 기반 기술 블록으로 분해한다.
- MVP와 이후 고도화 단계를 나눈다.

7. verifier / critic
- 중간 산출물과 최종 결과가 사용자 목표, 운영 제약, 기술 현실성에 맞는지 검증한다.

8. human-in-the-loop checkpoint manager
- 사용자 승인 대상 항목을 분리한다.
- stage 완료 조건과 rollback 가능 지점을 관리한다.

9. report generator
- 기획자용 요약과 기술자용 요약을 분리해 최종 결과를 생성한다.
- decision log도 함께 생성한다.

## stage 기반 상호작용 원칙
- 에이전트는 stage 단위로 작동하며, 각 stage는 명확한 산출물과 완료 조건을 가진다.
- 사용자가 현재 stage를 승인해야 다음 stage로 진행한다.
- high-impact 의사결정은 stage 완료 전에 반드시 사람 확인을 거친다.
- 미래 stage에서 새로운 제약이나 모순이 발견되면 이전 stage로 rollback할 수 있다.
- 최종 출력 전에는 verifier가 최신 사용자 의도와 stage별 합의사항을 다시 점검한다.
